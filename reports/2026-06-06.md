"建站 + 运营"工作流并到一个 chat 里。

**差异点**：和"AI 写文案 / AI 生图"那批 Shopify 周边不同，这是把
agent 拉成了 Shopify 控制台的同位替代品（catalog、collection、campaign
brief 都能直接由 agent 落地）。和 Shopify 自家 Sidekick 比，它跑在

**技术判断**：Manus 这次走的是"通用 agent → 垂直集成"路线，预计是
基于 Shopify Admin API + 一组结构化 tool 调用，模型层应该仍是 Manus
自家的多步 planner。需要绑定 Shopify 账号 + 付费 Manus 套餐，意味着
agent 的执行有 stateful 副作用（真的会改店）。

**信号价值**：是一个值得记下来的趋势点：上半年通用 agent（Manus、
Genspark、Devin 这一档）开始往"绑定具体 SaaS 账号、执行带副作用动作"
的方向走，下一步必然是 Stripe / HubSpot / Notion / GitHub 这种"账号
就是工作台"的 vertical connector 编年史。做通用 agent 框架的人现在
要想清楚：是做平台让别人接 connector，还是亲自下场做垂直绑定。

OAuth / 数据 schema 不一致——sleep / recovery / HRV / workout 几条
streams 各家定义都不一样，能干净标准化才有价值。比起做应用，更像
是在做"健康数据的统一 adapter 层"。

**信号价值**：MCP 生态正在跑通"垂直数据源 → 任意 LLM"的飞轮，

落点（个人持有数据、查询语义清晰、不涉及他人隐私）。值得跟踪的
是：等 Apple Health / Google Fit 自己出官方 MCP 之后，这种第三方
adapter 还有没有立足空间。

**定位**：一套"在 agent 回答之前先注入的 cognitive operation"。
开发者一次 API 调用就能拿到：要规避的失败模式、要执行的过程、要抑制
的捷径、要通过的自检项——本质上是把 prompt engineering 里反复手写
的 anti-pattern guardrail 工程化、可复用化。

**差异点**：和 LangChain guardrails、Anthropic 的 Constitutional AI、
NeMo Guardrails 这条线相比，Ejentum 的卖点是"按任务自适应生成

memory 四个 harness）。

**技术判断**：679 个 operation 这个数字本身有点市场口径，但
"adaptive mode 按任务重写每个 operation"意味着背后多半是
meta-prompt + retrieval。能否真的减少 drift / sycophancy 取决于
operation 的可执行性和 self-check 的可验证性，单看描述还判断不了。
适合自己接一下做盲评。

**信号价值**：agent infra 在分层：planner / executor / memory 之外
又冒出了"reasoning conditioning"这一层。这反映出业界对 raw LLM
reasoning quality 仍有不满，并且不再期待靠 base model 升级解决，转向
runtime-side 工程化干预。做 agent 框架的可以考虑把它当 "policy lint"
集成进去。

Pillow / Cairo 之类生成 1-bit PNG，调度走 cron 或自有调度。难点不大，

这个工程量比看起来稍多。

technology / e-ink 桌面派的人正在从"买 Daylight"转向"自己拼"。对
我们自己的工程方向有启发——很多"个人助手 / 日程 / 笔记"工具其实
更适合做成本地服务 + 多种前端（含 e-ink、终端、手表）。
