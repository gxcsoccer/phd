### 1. Latitude for Claude Code · ⭐⭐⭐⭐⭐
https://www.producthunt.com/products/latitude-4

**定位**：给 Claude Code 装一个「飞行记录仪」——逐 session 追踪完整的 system prompt、每一次 tool call、每个 subagent，以及每一轮的 token 成本。一行命令安装，trace 数据留在用户自己账户里，每周出一份用量报告。
**差异点**：现有的 LLM observability（LangSmith、Helicone、Langfuse）都是面向「你自己写的 agent」；Latitude 反过来观测一个你不拥有源码的黑盒——Claude Code 本身。切入点是「省钱 / 少撞 usage limit」这个非常具体的痛点，而不是泛泛的 tracing。
**技术判断**：不是套壳。能拿到 system prompt、subagent、per-turn token，说明它 hook 进了 Claude Code 的执行链路（很可能是 OTEL / 本地代理拦截），有实打实的工程。数据本地化是认真的卖点。
**信号价值**：Claude Code 已经从「一个工具」长成「一个值得做工具的平台」。当某个 agent 产品的 token 账单大到需要专门的 observability 层，说明这套工作流已经进入生产严肃度。做 agent 方向的人应该关注：可观测性会和 agent runtime 解耦，成为独立品类。

### 2. Apideck MCP Server · ⭐⭐⭐⭐⭐
https://www.producthunt.com/products/apideck

**定位**：一个 MCP server，把 200+ SaaS（会计 / CRM / HRIS / ATS 等）通过单一 endpoint 暴露给 AI agent，关键是带 scoped 读写权限和字段级脱敏——治理逻辑强制在 MCP 层执行。
**差异点**：和 Composio、Zapier MCP 这类「连接器聚合」最大的区别是它把 redaction / 权限当成一等公民。slogan 直接点名「别让 Claude 和 Codex 在客户的 SaaS 数据里乱跑」——卖的是约束，不是连接。
**技术判断**：有真东西。Apideck 本身是做 unified API 出身，把已有的 normalization 层包成 MCP 是顺势而为；field-level redaction 强制在 server 端意味着 agent 无法绕过，这是架构选择不是 prompt 约束。开源 + 兼容任意 MCP client。
**信号价值**：MCP 生态正在从「能不能连上」进入「连上之后怎么管」的阶段。agent 访问治理（权限、审计、脱敏）会是下一个明确的赛道。对做开发者工具的人：MCP 层的 policy enforcement 是一个还没被占满的位置。

### 5. CraftBot with Living UI · ⭐⭐⭐⭐
https://www.producthunt.com/products/craftbot

**定位**：一个通用 AI agent，它能在自己内部 build / import / evolve 自定义 app 和 dashboard，并且对这个「Living UI」的状态保持上下文感知——可以直接读、写、操作它。软件永远「未完成」，随需求增长持续改。
**差异点**：和 vibe-coding 类工具（Bolt、v0、Lovable）「生成一份代码交付给你」不同，CraftBot 的 UI 活在 agent 内部、agent 持续感知其状态。这是「agent-native software」而不是「agent 帮你写软件」——概念上更激进。
**技术判断**：难点在 agent 对 UI 状态的双向同步（读得到 + 改得动 + 不破坏）。描述偏愿景，没给架构细节，存在「demo 惊艳、深用即崩」的典型风险。但方向是真新颖的，不是套壳。
**信号价值**：和 Pipali（AI coworker，#12）一起代表一个趋势：软件交付物从「静态产物」转向「持续演化的、agent 托管的对象」。对做 agent 的人：这提出了一个真问题——agent 操作的到底是代码，还是一个有状态的运行时对象？
