### 1. Mina Meeting Assistant · ⭐⭐⭐⭐
https://www.producthunt.com/products/mina-meeting-assistant

**定位**：会议中实时介入的 agent——不是会后总结，是当场发言、调用工具、产出
artifact。把 "AI 在会议里" 从"录音转写"推到"第三个发言者"。

**差异点**：Otter / Granola / Fathom 这一代的卖点是会后摘要 + action items；
Mina 直接对标"会议中说话"。市面上类似形态只有 Read.ai 和早期的 Hume 试过
让 AI 说话，但都没把"在会议里跑 skill 产出文档/邮件"做成主卖点。可比对象不
多，框架是新的。

**技术判断**：能在会议里"实时响应"意味着低延迟 STT + LLM + TTS pipeline +
turn-taking 控制——这套组合不简单。描述提到"pulls context from your tools"
说明做了工具调用层。和 Sales call 形态绑定（CRM、knowledge base 调用）
基本是必然路径。是否原生模型还是基于 GPT/Claude realtime 不清楚。

**信号价值**：把 agent 从"异步任务执行器"推到"实时对话参与者"是范式跃迁。
和 OpenAI Realtime / Gemini Live 的能力曲线对得上——一旦这层 API 成熟，
"AI 作为会议参会者"会从猎奇变成默认。值得关注 Mina 怎么处理 turn-taking
和"何时插话"——这是该范式最难的产品问题，远比模型质量重要。

讨论度 28%（133c/475v），是头部里最高的——观众不是看热闹，是在追问能不能
真的做到。

### 2. Databox MCP · ⭐⭐⭐⭐
https://www.producthunt.com/products/databox

**定位**：Databox（BI 产品）把自家数据层暴露成 MCP server，能在 Claude /
ChatGPT / Cursor / n8n 里直接问业务数据。

**差异点**：和把 SQL/数据仓库接 MCP（PostgreSQL MCP、Snowflake MCP）不同，
Databox 把"语义层 + 业务上下文"暴露出来——指标定义、口径、跨源整合都已经
在 Databox 里建好了。LLM 拿到的不是裸表，是带语义的 metric tree。这正是
裸数据库 MCP 的痛点。

**技术判断**：Databox 已有的 metric / dashboard 抽象天然契合 MCP 的
resource + tool 模型。工程上不需要新发明，是把现有 API 包一层 MCP
spec——但产品决策很对：先做 metric query，再做 anomaly detection / report
generation 的话路径很顺。

**信号价值**：MCP 在 PH 上的曲线已经从"协议本身被讨论"过渡到"SaaS 把
MCP 当默认对外接口"。Databox 这种已经有自己 API 的成熟产品出 MCP
server，说明 MCP 不再是新创业公司的差异化点，是大公司的渠道接入策略——
等价于 2020 年的 Zapier integration、2022 年的 ChatGPT plugin。趋势已经
跨过 hype 期。

### 3. Dune Keypad · ⭐⭐⭐⭐
https://www.producthunt.com/products/dune-4

**定位**：三键的 Mac 物理键盘 + Claude 生成自定义脚本 + 社区脚本市场。硬件
是壳，核心是"用自然语言生成绑定到物理键的工作流"。

**差异点**：Stream Deck 是经典对照——但 Stream Deck 的脚本生态靠用户写
JavaScript / 调 API，Dune 的卖点是"对 Claude 说一句话就生成 binding"。
社区 marketplace 这步把 Raycast 的 extension 模型搬到物理外设上。
"context-aware" 指能识别当前 app 自动切换 binding 组——是 Stream Deck
没解决的功能。

**技术判断**：硬件本身不复杂（USB HID + 三个键 + 上下文检测）。难点全在
Mac 端 daemon：当前 app 识别、Claude API 调用、脚本沙箱执行、和系统权限
模型（Accessibility / Automation）打交道。Marketplace 又涉及代码审核
（任意 AppleScript 是巨大攻击面）。"自然语言生成脚本"在工程上是 Claude
Code 那一套模式——会有正确性 / 幻觉问题，但用户写的是 trigger，不是
agent。

**信号价值**：这是"AI + 新输入设备 + 社区生态"的少数高完成度组合。
Stream Deck 之后做了 10 年没人把它做成消费级 AI 入口，Dune 抓住了
"Claude 让人人都能写小脚本"这个窗口期。如果 marketplace 跑得起来，相当
于一个垂直版本的 Mac App Store——值得跟踪它的脚本审核机制和社区飞轮。

### 4. Typeahead · ⭐⭐⭐⭐
https://www.producthunt.com/products/typeahead

**定位**：在 Mac 所有文本输入框里做行内 AI 补全，纯本地模型，离线运行，数据
不上云。

**差异点**：和 Cursor / Copilot 不同，这是"系统级输入补全"——所有 app（邮件、
Notion、备忘录、浏览器）共享一个补全引擎。和 Apple Intelligence Writing
Tools 的对标是显然的，但 Apple 的功能是后置（写完再润色），Typeahead 是
predicting next token。Raycast AI / Rewind 都不在这个层面。

**技术判断**：在每个文本框里挂 IME-style 监听 + 调用本地模型 + 渲染 inline
ghost text——技术上 Apple Silicon native 是必须的（M-series ANE / Metal）。
本地模型应该是 quant 后的 3B-7B 量级。延迟控制是生死线（>100ms 用户就放弃
了），这意味着模型选择 + KV cache 复用 + speculative decoding 都得做。

**信号价值**：本地推理从"能跑 demo"进入"生产工具"——5/31 Clipto 是本地
多模态处理，今天 Typeahead 是本地 LLM 补全，5/30 Joanium 是本地 AI 工作
台。本地优先这一年从 niche 变成 PH 上能拿百票的常规形态，背后是 Apple
Silicon 推理生态成熟（MLX / Ollama / llama.cpp 链路稳定）。隐私是配角，
真正驱动是延迟和成本——离线意味着无 token 费、无网络抖动。

### 5. Tokenwise · ⭐⭐⭐
https://www.producthunt.com/products/tokenwise

**定位**：一行换 baseURL 就能接入的 OpenAI 兼容 LLM proxy，分析你的请求
找出"花得不值"的地方，提供一键替换。

**差异点**：和 OpenRouter / Helicone 不同——OpenRouter 是路由 + 价格表，
Helicone 是 observability。Tokenwise 主打"质量验证下的替换建议"：用你自
己的流量做 A/B，告诉你 GPT-4o 的某些请求换 Haiku 没区别，省 30%。这是
"benchmark on your traffic"，不是 public benchmark——这才是真痛点（public
benchmark 和实际业务流量差距巨大）。

**技术判断**：核心难点不是 proxy（很简单），是"质量 evaluator"。需要从用户
请求里学 task 类型 → 用 LLM-as-judge 或 embedding 相似度做无 ground truth
评估 → 给出可信结论。这套链路要做得不喧哗很难。

**信号价值**：LLM 成本优化今年是 picks-and-shovels 主战场之一。Tokenwise
的产品形态对——"一行 import 就上手 + 在你自己的数据上验证"，而不是让团队
跑 eval harness。如果做得好，会成为小团队默认接入的中间层。需要关注的
是它怎么解决"replay 流量做评估"的隐私问题（很多企业不会让 proxy 看到完整
prompt）。
