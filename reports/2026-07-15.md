### 1. Campus · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/flutterflow) | 443 votes · 110 comments

**定位**：为「人 + AI agents」协作而设计的共享工程工作空间——把仓库、终端、
项目知识、对话、agent 的历史工作全部沉淀到一个持久上下文里，替代散落在
Slack / Notion / 一次性 AI 聊天里的碎片。

**差异点**：市面上 Cursor / Devin / Codex 都在做「agent 直接改代码」，
但没人把「多个 agent + 多个人在同一个项目空间里持续协作」当作一等模型。
Linear 有 issue 但没 agent，Slack 有沟通但没工程上下文，Cursor 有编辑
但每次开局都要重放。Campus 押注的是「项目本身」是长期实体，agent 应该
像同事一样加入、离开、被交接。

**技术判断**：从描述看这是重工程投入——需要多用户实时协同的 filesystem
/ terminal 状态同步、agent 会话的可继承性、跨对话的持久上下文层。不是套壳，
和团队开发过 FlutterFlow（PH 上同一个 product 页面挂了他们，说明是老团队
的新押注）匹配。

**信号价值**：「project as first-class primitive for agents」是一个正在
被验证的方向。如果 AI agent 不是一次性任务执行者、而是长期驻留在某个
project 上的同事，那么 IDE / issue tracker / chat 都要重构。这条路径值得
跟踪：谁最先把「持久 agent 记忆 + 团队协作」跑通谁就有平台机会。

**差异点**：和 Zapier / n8n 这种「你写规则」的自动化不同，Agently 卖点是
「Agent 自己判断该做什么」；和 Salesforce Agentforce 那种「垂直 CRM 内置
agent」不同，它想做通用 orchestration 层。定位更接近去年 Adept / MultiOn
的野心，只是打的是 SaaS 后台。

**技术判断**：100+ connectors + 事件驱动 routing + 跨系统状态关联是硬工程。
但 tagline「今天的 AI 和明天 AGI 之间的那层」是危险信号，容易过度承诺。
真正决定成败的是：跨事件的记忆一致性、误路由的兜底、以及能否把
「connector 数量」转化成实际可用的自动化路径（Zapier 教训是 connector 多
但深度有限）。

**信号价值**：「event-driven agent routing」正在从 demo 走向产品化，是
所有做 agent 中枢的团队都必须回答的问题——如何把多个 workflow 收敛到
一个能自愈的执行层。值得关注它的定价和实际落地，但短期我不会押注宣传里
的 AGI 叙事。

成为可解释、可学习的招聘 agent。

**差异点**：这是我第一次在 PH 上看到把「Claude Skills」当作主要发行渠道

——用户在自己的 Claude workspace 里就能用，不用切上下文。这本质上是把
Claude 当作分发平台，类似 Slack app / Chrome extension 的策略但更深入
（因为 Claude 能主动调用 skills 而不是等你点开）。

**技术判断**：数据侧 Crustdata 一直在做（B2B 猎头数据 API），这次的
"技术"在于把 skill 拆分足够细致，让 Claude 能自然发现和组合（每次搜索
学习你的偏好、给出可解释的候选人排序、直接写入 ATS）。是标准的
data + Claude Skills 组合，没有黑科技，但把商业模式落地了。

**信号价值**：**极强**。这可能是「Claude Skills 作为分发渠道」的第一个
严肃产品案例。如果 Crustdata Recruiter 能证明 skills 商业闭环，会有大量
垂直 SaaS 跟进——skill 变成新的 API endpoint 品类，产品分发从「你来
我的网站」变成「我进你的 Claude」。做 AI agent 生态的团队都应该关注：
你的产品能不能拆成 skills？拆完之后怎么定价？

### 4. ccshare · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/ccshare) | 122 votes · 28 comments

**定位**：给 Claude Code 加了「多人协作」——6 位分享码 AirDrop 风格，
朋友从自己终端或浏览器加入，看到你完整屏幕，两个人都能敲键盘，本机 wifi
零配置或用免费 tunnel。

**差异点**：Claude Code 目前是单人体验。tmux + ssh 能做协作但门槛高、
需要账号。ccshare 直接把「分享 terminal 会话」做成了三秒钟的操作，且加上
了浏览器 fallback（对面不装 Claude Code 也能看）。这是 pair programming
从 IDE 时代（Live Share）迁移到 AI-native 终端时代的一个必然形态。

**技术判断**：本地 P2P + tunnel fallback 的经典分发架构。核心是 terminal
状态同步 + 输入冲突处理 + 加密。工程量不大但产品直觉锋利——发现了
「Claude Code 值得多人一起看」这个真实需求。

**信号价值**：确认了两件事——(1) Claude Code 正在从「个人工具」演化为
「团队协作载体」，围绕它会长出一堆生态工具；(2) terminal 作为 AI 交互界面
的地位在加固，反而是 IDE / web UI 层出现工具真空。做开发者 AI 工具的
方向：与其做新 UI，不如做 Claude Code / Codex 周边的胶水层。

### 5. QuickQuill · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/quickquill-2) | 110 votes · 44 comments

**定位**：纯本地、on-device 的 Mac 会议助手——完全不联网也能录音 / 转写 /
生成 summary / 实时字幕带翻译 / 全局 push-to-talk 听写。无账号、无订阅、
一次性 $49，dictation 永久免费。

**差异点**：市面几乎所有 meeting AI（Fireflies / Otter / Granola）都是
云端处理，需要 bot 加入或上传音频。QuickQuill 完全反着来：本机推理、
关掉 wifi 也能跑（demo 视频就是这么录的）、没有 bot 打扰对方。这条路径
只对 Apple Silicon + macOS 26 (M-series 新一代) 用户成立——它把
Apple Neural Engine 的算力当作免费基础设施用。

**技术判断**：核心是本地 ASR + LLM summarization。macOS 26 大概率带了
新版 Apple Intelligence 模型可用，QuickQuill 是它的第一批利用者。工程
挑战是实时字幕的延迟控制和翻译质量；商业模式的赌注是「一次性买断」在
订阅疲劳时代能否比 Otter 类竞品更受欢迎。

**信号价值**：**local-first 是本周最清晰的信号之一**。GPT-5 之后云端能力
过剩，用户开始重新在意 privacy + offline，Apple Silicon 的推理算力让
「无云也能跑」在真实应用里成立。开发者工具方向可以借鉴：不是所有 AI
产品都要绑定 GPU 云端；on-device inference 是一个正在扩大的机会窗口，
尤其是隐私敏感场景（医疗 / 法律 / 会议 / 输入法）。
