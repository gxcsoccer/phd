### 3. zero.xyz · ⭐⭐⭐⭐
https://www.producthunt.com/products/zero-xyz

**定位**：给任何 CLI agent（Claude Code、Codex、Gemini CLI、OpenClaw 等）插一根总线，让它能调用约 8000 个第三方服务的 API/SDK，不用使用方配置任何 API key。

**差异点**：和 Composio 几乎正面对线。Composio 走的是"开发者集成中心"路线，需要用户在自己平台上接服务。zero.xyz 把交付形态做成"CLI agent 一行命令就能直接用"——把工具注册从"agent 工作前的配置"变成"agent 工作中的发现"。这是从静态工具表向动态工具发现的范式迁移。

**技术判断**：8000 个集成不太可能都是手写适配——大概率是 OAuth proxy + OpenAPI/schema 自动转 MCP-style tool definition 的工厂。免 API key 的方式只可能是"zero.xyz 持有用户授权，agent 调它的 proxy"。这意味着 zero.xyz 是把自己变成所有 agent 工作的中央 OAuth/Token broker——商业模式和卡位都很激进。

**信号价值**：MCP 协议落地一年后，第一个真正在做"通用工具发现层"的产品。如果做成，agent 工程的重心会从"为 agent 写 N 个 tool"迁移到"为 agent 配一根总线"。对自建 agent stack 的方向有直接威胁——你也许不需要为每个集成单独维护 MCP server 了。

**关注度**：⭐4——votes（293）+ comments（73）双高，ratio 0.25，社区认真在讨论"和 Composio 比怎么样"。

### 5. Chunk sidecars · ⭐⭐⭐⭐
https://www.producthunt.com/products/chunk-sidecars

**定位**：在 agent 写完代码 commit 前、push 到共享 CI 之前，在本地跑一个"CI 的微型镜像"做范围内构建/测试。让 agent 在 27 秒内拿到反馈，而不是 5 分钟后从远端 CI 拿失败结果再重试。

**差异点**：这是第一个把"CI 反馈环路"专门做成 agent 工作流原语的产品。现有 sidecar 类产品（Nx、Turborepo cache）是给人用的；CircleCI/GH Actions 是远端运行的。Chunk 解决的是 agent 特有的痛——agent 的 context 在 push 那一刻就丢了，等 CI 反馈回来要重新组装。

**技术判断**：核心技术是"自动检测栈 + 范围检测"——根据 diff 决定跑哪些测试和构建。本质是 affected-graph 计算 + 本地容器化执行。"3x–5x fewer tokens in retry loops"是因为反馈快+局部，agent 不需要重新读整个 repo。明显是从真实 Claude Code / Cursor 使用痛点反推出来的工程。

**信号价值**：表明 agent 工作流的瓶颈从"模型能力"转向了"反馈延迟"。当一个 agent 一天能跑几十次小迭代时，CI 的 5 分钟从可接受变成致命。这类"本地 fast-feedback for agents"产品会越来越多——预计接下来会有针对类型检查、lint、E2E 的同形态产品。

**关注度**：⭐4——votes 不高（71）但 CircleCI 背书 + 解决的问题非常真。对做 agent loop 的人来说是直接可用工具。
