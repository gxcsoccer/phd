prototype 这一类用例的零门槛入口。

**信号价值**：和今年看到的 v0、bolt.new、replit-agent 一类是同一个方向：
"开发的最后一英里 == 0 点击"。Vercel 在他们的护城河（CDN + serverless）
上加这个入口，会进一步把"我有一个产物想给别人看"的默认动作锁死在他家。
对开发者工具方向的提示：摩擦砍到 0 之后，工具的竞争维度从"功能"切到

### 3. Prometheus by Firecrawl · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/extract-by-firecrawl) | Firecrawl (YC S22)

**定位**：用自然语言描述你要从 web 上抓什么数据，agent 自动写 Firecrawl
代码去抓，并且页面变了之后会自动维护。"Forward Deployed Agent for web
data."

**差异点**：和市面上 web scraping SaaS（Apify、Bright Data、ScrapingBee）
不同：那些卖的是"代理 + 渲染 + selector 维护服务"，Prometheus 卖的是
**让 agent 自己写 scraper 代码并对它负责**。和 Browserbase、Stagehand 这
类 "AI 操作浏览器" 路线也不同——后者每次都让 LLM 实时操作浏览器（贵且
慢），Prometheus 是一次 codify、长期复用、变了自动修。

**技术判断**：核心架构猜想 = LLM 把用户描述编译为 Firecrawl SDK 代码 +
持续 diff 监控 + 失败时回流给 agent 重写。亮点是把"维护成本"这个 scraper
最痛的环节交给 agent 持续运行。Firecrawl 既然自己卖底层 scraping
infra，做这一层是顺理成章——他们对"页面什么时候坏掉"的信号最清楚。

**信号价值**：明确把 agent 从"一次性问答"推进到"长期运行的 software
component"。这是值得反复看的 pattern：**agent-as-codegen，不是 agent-as-
runtime**。让 LLM 生成代码、让代码跑 production 工作量、让 agent 只在
代码失效时介入——比"LLM 在 hot path 上每次都跑"成本低几个数量级。
做 AI agent / 自动化产品的方向应该认真学这个分工。

SAM-Lite 类模型）+ on-device classifier 或多语言 embedding。能跑到
iPad/iPhone 上说明模型已经压缩到 <500MB 量级。kids 这个用户群体决定了
不上云是正确的产品决策（COPPA、GDPR-K）——这不是营销话术，是合规
路径。

**信号价值**：on-device AI 这个方向今年明显在加速（连续两天都有强信号；
