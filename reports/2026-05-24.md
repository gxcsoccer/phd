### 1. ModelHub · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/modelhub) · 305 votes / 40 comments / r=0.13

**定位**：Mac 菜单栏里的本地 LLM 控制中心。把 HuggingFace 模型发现、下载、版本管理这件事从「浏览器 + 终端 + 一堆模型卡片来回切」收拢成一个原生 GUI，然后把模型喂给 Ollama / MLX / LM Studio / llama.cpp。

**差异点**：不去和 Ollama / LM Studio / Jan 抢「跑模型」的位置，而是补在它们上层做「发现 + 库管理」。这是个我自己用 Ollama 一直觉得别扭的痛点——`ollama pull` 之前我得先去 HF 翻 model card、判断量化版本、对照 system RAM 决定下哪个 GGUF，整个流程没有任何 GUI。

**技术判断**：原生 Cocoa 菜单栏应用 + 多 backend 适配层。难点不在客户端实现而在「让 HF 各种命名混乱的量化版本可以一键映射到 Ollama / MLX 期待的格式」，这是 librarian 类工作，做好了价值很高，做不好就只是个 HF 浏览器。从 40 评论里能看到作者在回答具体的 backend 选择问题，说明这层抽象不是嘴上说说。

**信号价值**：本地 LLM 工具链终于开始出现「Day-2 ops」层——之前所有项目都在抢「能跑」这个 Day-1 位置，现在开始有人做发现、管理、监控。如果未来 6 个月本地推理在开发者侧持续渗透，这种 hub 类工具就是 Ollama 之后的下一波分润位。值得长期关注。

### 2. Edgee Fallback Models · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/edgee) · 178 votes / 21 comments / r=0.12

**定位**：Claude Code 的高可用代理层。Anthropic 抖动 / 配额耗尽 / 价格太贵时，无缝路由到 Kimi K2.6 / Gemma / GLM / Qwen，或者你自己的 Bedrock / Vertex / Azure 账户，对 Claude Code 客户端完全透明。

**差异点**：和 OpenRouter / LiteLLM 比，差别在两点。一是只针对 Claude Code 这一个客户端做专属适配（不是通用代理），二是把 fallback 作为头等公民——OpenRouter 的 fallback 是手动列表，这里更接近 SRE 意义上的「primary / secondary」健康检查路由。「Claude Code that never stops」是把基础设施话语接管到了 agent 工具上。

**技术判断**：本质是个针对 Anthropic Messages API 的反向代理，加上一层模型能力等价映射（tool use、prompt caching 这些怎么在 Kimi/GLM 上模拟）。如果 mapping 不严肃，fallback 之后 Claude Code 的 agent loop 会立刻崩——会议室里说说容易，工程上难。21 条评论里如果有人在追问 prompt caching 的兼容性，那就是真的；如果只是「太需要了」，那就是 demo。

**信号价值**：这是我一直预期会出现的一种产品——**给单一 agent 客户端做 SRE-级别的可靠性代理**。这意味着 Claude Code 已经被一部分团队当作「生产关键路径上的工具」，开始有人愿意为它的 uptime 付钱。如果你在做 agent 产品，这是一个值得借鉴的「不要自己做 fallback，让别人做」的市场切口。

### 3. Freu AI · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/freu-cli) · 286 votes / 16 comments / r=0.06

**定位**：Mac 桌面 app 自动化 agent。用自然语言描述任务，agent 看一次 UI 后编译成一份「跨 app 的确定性 DSL」工作流，之后本地重放，不再每次都调 LLM。同时开源了 `freu-cli`（浏览器自动化引擎）。

**差异点**：和 Anthropic Computer Use / OpenAI Operator 走的是相反路径——后者每一步都让 LLM 实时看屏幕、决定下一步动作（贵、慢、不稳）；Freu 是「LLM 编译一次，DSL 重放无限次」。这个范式我之前看到 LangGraph 和 Adept 团队提过类似想法，但产品化做得最干净的是这个。「$0 recurring run cost」是这个范式最直接的卖点。

**技术判断**：编译式 agent。难点在 DSL 的鲁棒性——UI 一旦变（位置 / DOM / 控件 ID），DSL 失效，agent 需要自动重编译。tagline 强调「no brittle coordinates/selectors」说明他们大概率用的是视觉 + 语义混合的元素定位，不是 XPath / accessibility tree 直读。开源 `freu-cli` 是个聪明的获客 + 信誉建设动作。

**信号价值**：**「Compile-once agent」是 agent 工程化的一个重要分支**，比每一步都问 LLM 的 ReAct loop 经济性高一个量级。如果你在做 agent 工具，应该认真评估这种「LLM as compiler，runtime is deterministic」模式在你场景下是否成立。这是今天最值得做技术深挖的一个产品。

### 5. JellyNet · ⭐⭐⭐
[PH](https://www.producthunt.com/products/jellynet) · 18 votes / 6 comments / r=0.33

**定位**：LLM API 配额二级市场。供给方贡献闲置的 40+ LLM 提供商 API key 赚 USDC，需求方用一个统一 API key 拿到自动 failover 的池子。AI agent 通过 x402 协议直接按调用付费，无需账号或 API key 管理。

**差异点**：把「LLM 调用」当作可以做市场清算的标准化商品。这个想法 OpenRouter 已经在做了，区别是 JellyNet 把「闲置 enterprise 配额」也接进来，并通过 x402（Coinbase 发的 HTTP 402 微支付标准）让 **agent 自己能付钱**——不用人类先去注册和充值。

**技术判断**：x402 + USDC on Solana 是这个产品真正新的部分。x402 的设想是「HTTP 协议层上原生的微支付」，agent 收到 `402 Payment Required` 就自动用 USDC 付，再发请求。这等于给 agent 经济一个底层结算层。链路上的工程复杂度其实在「key 信誉评分 + 路由抗欺诈」这块，做不好就是劣质 key 全部被挑出来。

**信号价值**：今天票数最低的入选项，但**信号最稀缺**。如果你在做 agent，「让 agent 自己付费消费 API」这件事是个被低估的基础设施位——大多数人还在想怎么让 agent 用 tool，没想到 tool 是要花钱的、人类 KYC 路径很难给 agent 用。需要观察的是 x402 是否真能成事；如果半年后还有 5+ 项目采用，这就是个新 stack 层。注：用了 USDC/Solana，但属于支付轨道而非投机性 crypto 产品，未触发本日报的过滤红线。
