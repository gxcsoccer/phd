### 1. Miora · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/miora-2) · 578 votes / 126 comments

**定位**：一块可编辑的创作画布，附带一个跨会话记住"你的品味"的
agent；一次输入 idea，输出多模态素材（图 / 文 / 视频片段等），
过程被自动沉淀成可复用的 "Skill"。

**差异点**：目前市面上的 AI 图像/多模态工具主要沿两条路走——
Midjourney/Ideogram/Adobe Firefly 的"输入 prompt 出结果"，
以及 Freepik Pikaso/Canva Magic 的"模板 + 编辑"。Miora 把两者塞进
同一块画布，再加一层长期记忆（"agent memory"→"Skill"）。
类比 Cursor 的 "@rules" 或 ChatGPT Memory，但落到创作品味 /
风格约束上——同类里最接近的可能是 Krea 的 Style Reference，
但 Krea 是单次绑一张 ref，Miora 强调自动累积。

**技术判断**：从描述看，"Skill" 大概率是把用户历次调整（正/负例、
风格向量、常用元素）落成一个可召回的 embedding + rule pack；
"editable canvas" 与"多模态"合在一起，底层多半是外包若干
generation API（Gemini/Flux/Sora 等）+ 自研的 diff-based canvas 组
合调度，不是自训模型。工程主要在 orchestration / memory retrieval 层。

**信号价值**：**"Memory as product"** 这件事在 07-12 已经不是个案。
Top 5 里同一天出现三家从不同侧面做记忆（Miora 创作、Second Brain 通用
context、FetchSandbox 测试 case），加上下沉候选 Dcyde（决策记忆），
说明"给 AI 装长期记忆"这个 wedge 从概念阶段进入到产品差异化阶段。
对做 AI agent 的团队值得思考：memory 层是护城河还是通用 primitive？

### 2. JustVibe · ⭐⭐⭐
[PH](https://www.producthunt.com/products/justvibe) · 563 votes / 88 comments

**定位**：一个"搜索即执行"的入口——输入 `plan my 5 day trip to tokyo`
就在浏览器里生成一个可交互的行程规划 app；若已有的 app 都不匹配，
就现场用 LLM 生成一个，无需代码，可以永久保存并分享。

**差异点**：这是 v0/bolt.new/Lovable 家族的又一个"搜索包装"变体。
v0 是"聊天 → app"，JustVibe 是"搜索 → app"，UX 门槛更低。
真正的对标其实是 Google/Bing——用生成式 UI 替代传统搜索结果页，
逻辑接近 Perplexity 的 "generative UI" 演化路径，但 JustVibe
更彻底：不给回答，直接给一个可用的小应用。

**技术判断**：从"分钟级生成 + 永久保存 + 可分享 link" 推测：一个
LLM（GPT-5 / Claude 5 级别）+ 预置的 UI component library（可能是
React + shadcn 之类）+ 静态站点托管。核心难点是把 prompt 稳定
落到"能跑的一段 UI + state"，套用度可以做很高。
架构与 Vercel v0 相似，工程不算深，但 UX 想法有意思。

**信号价值**：**generative UI 从"给开发者用"往"给消费者用"迁移**
的一次尝试。若真能跑通（真实使用中的 app 质量、留存），代表
"每个问题都值一个一次性 app" 这个假说被验证——但也可能是又一个
demo-fresh、用户回不来的产品。观察点是 30 天后的 DAU / 应用复用率。

### 3. FetchSandbox · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/fetchsandbox) · 413 votes / 84 comments

**定位**：给 AI coding agent（Cursor / Claude Code / Windsurf / Codex /
VS Code）用的 API 集成沙箱。通过 MCP 暴露 60+ 三方 API
（Stripe/GitHub/Clerk/Resend/Twilio/OpenAI 等）的可控 mock 环境——
不仅返回 200，还能模拟 webhook、重试、异步状态转移、失败场景，
并"记住"重现过的 bug。

**差异点**：过去 API mock 类工具（Mockoon、Beeceptor、WireMock）
面向的是开发者，目标是节省 staging 环境成本。FetchSandbox
把 target user 换成了 **AI agent**，多做了两件事：
(1) MCP 接口，agent 可以直接调；
(2) 保留 failure history，让 agent 生成的测试能复现过去出过的问题。
最接近的对手是 Prism (Stoplight)、Postman Mock Server，但都还没把
MCP + agent 上下文当第一优先级。

**技术判断**：MCP server + 一堆预置的 API 语义模型 + 事件回放引擎。
"remembers what breaks" 意味着有一层持久化的 case store，很可能
按 endpoint × request shape 索引失败模式，agent 每次跑测试会先查
历史再决定要造什么用例。60+ API 的覆盖听起来重，但如果只是
"最小 mock + 已知 quirks 表"，量级不夸张。

**信号价值**：**"面向 AI 而不是面向人的开发者工具"** 是今年一个
清晰的赛道分化。FetchSandbox 显式把 MCP 当第一等接口而不是
"未来 nice-to-have"。做 dev tool 的团队需要问一遍自己：如果我的
API 是给 Claude Code 而不是给工程师用的，产品长什么样？很多现有
UI-first 工具都要被重做一遍。今天最有借鉴价值的一条产品线。

### 4. Second Brain for AI v2 · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/second-brain-cloudflare) · 365 votes / 96 comments

**定位**：跨 LLM 客户端（Claude / ChatGPT / Cursor / Codex + 任意
MCP 客户端）的共享 memory 层——存项目、人、决策、偏好。v2 新增
自动关联相邻记忆、区分"已敲定 vs 草稿 vs 陈旧"、以及在 Cloudflare
账户下自托管。开源。

**差异点**：Mem0、Zep、Letta 是同赛道的通用 memory backend，
但都要求集成方主动调用。Second Brain 把入口做在了 **MCP 客户端**
一侧——只要 host 支持 MCP，就能直接接进来，用户不用改任何客户端
配置。加上"self-hosted on Cloudflare Workers/D1/KV" 的部署方式，
数据主权 + 零运维两条都占了。

**技术判断**：核心是 memory graph（关联抽取 + 时序衰减）；v2 的
"distinguishes settled decisions from drafts and stale context" 意味着
它在做 state classification 而不仅是 embedding 存取——大概率有一层
LLM-based classifier 每次写入时打标签。Cloudflare 全家桶部署路径
很聪明：Workers + Vectorize/D1 就能撑起一个便宜的 memory server。

**信号价值**：MCP 生态在快速形成"通用中间件"层。上周已经看过几个
MCP tool（router、guard），今天看到 memory。**MCP 正在长成一个
横向可组合的 agent OS**——每个 primitive 都会有开源实现和 hosted
版本竞争。做 agent 应用的人，应该开始把 memory 当 external
dependency 而不是自己造。

### 5. ServiceBeard · ⭐⭐⭐
[PH](https://www.producthunt.com/products/servicebeard) · 159 votes / 27 comments

**定位**：把客服邮箱同步进 issue tracker（GitHub / GitLab / Linear），
让你的现有工作流兼作 helpdesk，不用再买按座位付费的
Zendesk/Intercom。开源，走标准 IMAP/SMTP。

**差异点**：这条路早有玩家——Plain、HelpScout、Freshdesk——但都是
"独立 helpdesk 系统"。ServiceBeard 反其道而行：不做 helpdesk UI，
只做双向同步适配器，把 issue tracker 当 CRM 用。对已经在 Linear /
GitHub 里跑 workflow 的小团队是明显更轻的方案。类似思路的还有
Front + GitHub sync，但 Front 本身也是付费产品。

**技术判断**：IMAP 拉信 → 结构化解析 → 对应 issue tracker 的 REST
API 双向同步 + 状态映射。技术不复杂，但邮件线程 → issue 评论的
映射、多次回复的去重、attachment 处理这些边界情况会决定用户体验。
开源 + self-host 意味着数据不出租户，适合有安全要求的中小团队。

**信号价值**：**"不做产品，做 adapter"** 是开源工具的一个稳定占位
策略。当 Linear、GitHub Projects 这类工具越做越像"通用协作后端"
时，adapter 层的机会窗口就打开了。同类思路值得关注：把 Notion /
Airtable / Linear 当后端的一切轻量前端。
