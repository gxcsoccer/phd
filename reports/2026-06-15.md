### 1. Novu Connect · ⭐⭐⭐⭐
https://www.producthunt.com/products/novu

**定位**：把"AI agent → 用户"的通信层抽出来做成一个 SDK——你写 agent
逻辑（任意模型、任意框架），Novu 负责把 agent 的消息双向跑通 Slack /
Teams / WhatsApp / Telegram / Email，包括身份解析、threading、路由、
渠道适配的格式化。开源。

**差异点**：表面上是"agent 版的 Twilio + Courier"，但真正的卡位点是
"two-way + 身份归一"——目前主流框架（LangGraph / CrewAI / OpenAI Agents
SDK）都假设交互发生在自家 chat UI 里，要把 agent 落到客户已经在用的
messenger 里，每家自己写适配是一周到一个月的工程量。和 Novu 母公司原本
的 notification orchestration 形成自然延伸——通知是单向，agent 通信是
双向 + 有状态，差一个量级但走的是同一套渠道。

**技术判断**：核心是 inbound webhook 归一 + 出站 channel adapter + 跨
渠道 identity resolution（同一个人在 Slack 和邮箱是同一个 thread）。
不是 wrapper——这套 plumbing 极琐碎，且每个渠道的限速 / 富文本 / 附件
规则都不一样。开源策略意味着想把"agent 通信层"做成事实标准，而不是
靠托管赚钱。403 票 + 63 评论（评赞比 0.156）说明 dev 群体真的痛。

**信号价值**：这是 6-14 报告里"MCP 解决工具可插拔、还缺什么"列表的
答案之一——agent 的 transport layer 现在还是各家自造轮子。和 6-10 那条
"agent infrastructure 正在分层"的判断对得上：编排（Hive 类）/ 观测
（PandaProbe / Conan）/ 通信（Novu Connect）/ 验证（AEVS）四层正在
各自跑出独立玩家。如果你做 AI agent 方向，"通信层标准化"会很快从
nice-to-have 变成默认依赖——和 6 年前的 Twilio 之于 SaaS 同构。

### 2. AEVS · ⭐⭐⭐⭐⭐
https://www.producthunt.com/products/aevs-by-fetch-ai

**定位**：给 AI agent 装"行车记录仪"——drop-in SDK 把每一次 tool call
的输入 / 输出 / 状态 / 时序记成可验证、tamper-evident 的执行收据。
不依赖聊天历史，不依赖脆弱日志。出品方是 Fetch.ai。

**差异点**：和 PandaProbe（同日 Top 4，agent observability 托管）/
LangSmith / Langfuse 这些 trace + eval 工具的本质区别在"verifiable"
三个字——前者是"团队内部看 agent 做了啥"，AEVS 是"对外能证明 agent
做了啥"。这两个需求层完全不同：observability 服务于内部 debug，
verification 服务于审计 / 合规 / B2B SLA 履约证明。和 EigenLayer
那种 web3 AVS 比，AEVS 走的是 SDK 路线而不是 protocol——更落地，更
适合实际生产。

**技术判断**：要让收据可验证，关键不是 hash 链本身，而是"何时签 + 谁签
+ 输入侧的真实性"。tool call 的 input/output 如果可以被本地篡改后再签，
那链什么都证明不了。Fetch.ai 在 agent identity / on-chain settlement
上有底层积累，AEVS 多半是把已有的密钥层和签名链接到 SDK——这才是
"drop-in"能做成 tamper-evident 的前提。如果只是"把日志 hash 存一份"，
就退化成普通审计 log。

**信号价值**：agent 进入生产后，"agent 真的做了它声称的事吗"会变成
合规 / 保险 / SLA 的核心问题——尤其是 agent 操作真实账户（订机票、
付款、改数据库）时。这条赛道的 TAM 长在企业合规预算里，不在 dev 工具
预算里。值得 watchlist：6-12 那条"agent insurance / liability"的预判
开始有基础设施了——AEVS 是"可投保的前置条件"。今天评赞比 0.203
是 Top 10 里第二高，dev 群体已经把它当成认真的工程产品来评。

### 3. MiMo Code · ⭐⭐⭐⭐
https://www.producthunt.com/products/mimo-3

**定位**：基于 OpenCode fork 的开源终端 coding agent，主打"显式长期
记忆架构"——用一个独立的 checkpoint subagent 来管理超长 context，
在 sandbox 里执行工作流，按计划做"自我维护"（presumably 重新整理
memory / 修正错误总结）。

**差异点**：和 Claude Code / Codex / Aider / Cline 的 context 处理
方式不同——后者要么靠模型自身 200K context 硬扛，要么靠简单的
file-list + recent-edits 滚动窗口。MiMo Code 把"记忆管理"提到一等
公民，单独一个 subagent 在主线之外维护检查点。这接近 Letta / MemGPT
的思路（长期 / 短期 memory 分层），但落到 coding agent 这个具体场景。
和 OpenCode 的差异是加了 memory 子系统 + scheduled maintenance，不是
表层 UI 差别。

**技术判断**：checkpoint subagent 的设计如果做对了，能解决 coding
agent 跑长任务时"忘了自己 30 步前为什么改这个文件"的核心痛点。但
有两个工程难点没法从描述看出来：(1) checkpoint 的触发条件是固定
步长还是基于 token / 事件？(2) 主 agent 怎么消费 checkpoint——直接
拉进 context 还是按需检索？这两点定生死。开源 + 基于 OpenCode 意味着
可以验证。

**信号价值**：6-14 报告里 Memoriq 那条"portable AI memory"是用户侧，
MiMo Code 是 agent 工程侧——两件事其实指向同一个判断：**记忆层正在
从模型内部 emerge 成独立的工程模块**。下一阶段 coding agent 的核心
差异化会从 "context window 大小" 转向 "记忆管理质量"。如果你做
agent framework，memory 子系统会和 tool 子系统并列成为框架的核心
抽象——这是个具体可落地的方向。
