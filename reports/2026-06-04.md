### 1. Google Gemma 4 12B · ⭐⭐⭐⭐⭐

[PH](https://www.producthunt.com/products/gemma-4-12b)

**定位**：Google 开源的 12B 多模态模型，**没有独立 encoder**，文本/视觉/音频
在同一个 transformer 主干里走，目标硬件是 16GB VRAM 的本地机器。

**差异点**：与 Gemma 3、Llama 3.2 Vision、Qwen2-VL 这一代相比，去掉
encoder 是工程上的硬切换——之前多模态几乎都靠"独立视觉/音频 encoder
+ adapter → LLM"的拼接方案。encoder-free 意味着 token 化和 modality
对齐被压到模型内部，理论上推理路径更短、本地部署内存占用更可控。
对标的不是闭源 multimodal API，是"在 RTX 4080/5080 / 高配 Mac 上跑本地
agentic 应用"这条线。

**技术判断**：从 tagline 列出"encoder-free"和"16GB VRAM"这两个指标
看，Google 显然在把 Gemma 系列重新定位成本地 agent 的基础模型，不是
做给云推理的。这是过去一年开源多模态最有架构含金量的更新之一。

**信号价值**：本地多模态 agent 的基础底座在变好。半年前做"本地跑的
GUI agent / 屏幕理解 agent"还要拼多个模型；现在一颗 12B 就能覆盖。
对自己做开发者工具或 agent 产品的方向意味着：cloud-only 的多模态
功能正在丢护城河，**privacy / 离线 / 成本** 这三条线的产品有了能
真正落地的基模。

**关注度**：⭐⭐⭐⭐⭐ — 评论只有 7 条（PH 社区对模型本身不擅长讨论），
但本质级别的更新，不该用 comments 评估。

### 2. Keen Code · ⭐⭐⭐⭐

[PH](https://www.producthunt.com/products/keen-code-a-cli-coding-agent)

**定位**：用 Go 写的开源 CLI coding agent，自称"由 coding agents 自己
构建"，repo 里完整保留 prompt 和设计轨迹。两个技术点：**turn memory**
压缩多轮上下文，**MCP servers 映射成 lazy-loaded Skills**。

**差异点**：和 Claude Code / Aider / OpenCode / Continue 这一批同类
比，Keen Code 的两个技术选择都很反共识：
- "MCP 即 Skills"：现在主流 MCP 客户端是开会话就把所有 MCP server 的
  schema 全塞进 system prompt，多 MCP 场景下 token 浪费非常严重。
  Keen Code 把每个 MCP server 包成 Skill，按需加载——这是 Claude 在
  10 月才推的 Skills 范式，Keen 直接拿来当 MCP 的取代物。
- turn memory：不是 summary，而是 turn-level 的上下文压缩，思路接近
  Aider 的 repo map 但作用于历史对话。

**技术判断**：Go 实现 + 把 prompt 公开成 repo + 明确的上下文优化策略，
说明作者关心的是"长会话场景下的 token 经济"，而不是另做一个
demo-friendly 的 UI。是同类里少见把"context engineering"当卖点的
产品。

**信号价值**：CLI coding agent 进入分化期——前期都在拼"能不能用"，
现在开始拼"长会话下贵不贵 / 准不准"。Skills 范式从 Claude 一家
专属变成更通用的 agent 工具组织方式，这个收敛速度比预期快。

**关注度**：⭐⭐⭐⭐ — 投票中位，但目标读者就是 agent 工程师；
prompt 公开 repo 这一点本身就是同类里能学到东西的素材。

### 3. Kai for Chrome · ⭐⭐⭐⭐

[PH](https://www.producthunt.com/products/kai-for-chrome)

**定位**：把 Whisper Base 跑在 Chrome 扩展里，靠 **WebGPU + WASM** 做
Meet/Zoom/Teams 的本地转写和总结。不要账号、不要上传、不进会议室。

**差异点**：现有的会议转写要么是云 STT（Otter、Fireflies、Granola）
要么是桌面应用（Plaud、Superwhisper），**浏览器扩展跑完整 Whisper Base
的可能是头一份**。技术路径上，Web LLM 项目（mlc-ai）和 transformers.js
都在推这个方向，但真正商业产品落到 PH 上、目标用户是普通会议软件
用户的还不多。

**技术判断**：Whisper Base 大约 74M 参数，量化后能塞进 WebGPU 是
WebLLM 生态成熟的标志。难点不在跑 Whisper，是把"实时音频流 → WebGPU
推理 → 整合 Meet/Zoom DOM"这一整套工程化封装好——Morgen 团队明显有
做完整 OS 集成的能力。

**信号价值**：浏览器变成 on-device AI 的部署目标。这条线和 Gemma 4 12B
是同一个趋势的两端——硬件够强了，大家终于愿意把推理搬回客户端。
做 agent 产品时，"必须有云推理"开始不是个无可争辩的前提。

**关注度**：⭐⭐⭐⭐ — 90 投票 / 15 评论。技术层面是过去几个月最干净
的 WebGPU 推理落地案例。

### 4. Boxes.dev · ⭐⭐⭐⭐

[PH](https://www.producthunt.com/products/boxes-dev)

**定位**：**每个 Claude Code / Codex 会话一个云端 VM**，跨手机/桌面接入，
专为 agentic coding 准备的 sandbox infra。

**差异点**：和 Cursor Background Agents、Codex Cloud、Devin、Replit
Agent、E2B、Daytona 比：
- Cursor / Codex Cloud：每家 IDE 厂商自己的封闭后端
- E2B / Daytona：通用的 sandbox-as-a-service，不预装 agent
- Boxes：把"每个 chat session 独立 VM"做成第一公民，agent CLI（Claude
  Code、Codex）开箱即用，重点在**多 session 隔离 + 手机接入**

这是把"我在通勤路上能继续昨晚那段 vibe coding"做成基础设施的产品。

**技术判断**：从 tagline 看是 firecracker / kata-style 微 VM + agent CLI
预装 + Web/Mobile UI 接入层。难点不在虚拟化，是 session 状态的持久化
和跨设备同步——多人都在尝试，Boxes 是少数明确把"每个 chat 一台
机器"作为产品边界的。

**信号价值**：agent dev env 这个赛道开始把"per-session VM"当默认。
长期看会冲击 IDE 厂商对开发环境的所有权——开发环境从"装在我电脑上"
变成"挂在 agent session 上"。

**关注度**：⭐⭐⭐⭐ — 22 条评论显示开发者关心实操（限速、价格、
session 持久化）；同赛道里值得追踪定价模式如何演化。

### 5. Intelligent Terminal · ⭐⭐⭐⭐

[PH](https://www.producthunt.com/products/microsoft-terminal)

**定位**：Windows Terminal 的开源 fork，**原生集成 ACP（Agent Client
Protocol）兼容 agent CLI**：agent 状态栏、context-aware agent pane、
自动错误检测、session 管理、命令面板 prompt。

**差异点**：和 Warp / WezTerm（带 AI 命令补全）/ Hyper 比，最关键的
差异是 **ACP 协议而不是某家专属集成**。Warp 把 AI 锁在自己的后端，
WezTerm 还在用插件方式接入；Intelligent Terminal 选择把 agent 当成
任何 ACP-兼容 CLI 都能挂进来的一等公民。

**技术判断**：fork 微软的 Windows Terminal 本身是工程量很大的选择——
拿 C++/WinUI 代码库改并维护住，说明作者对终端协议层有深入理解。把
ACP 落到 Windows 平台是有意义的——这个协议大半年还停留在 macOS /
Linux CLI 里。

**信号价值**：终端这个最古老的开发者 UI 正在被 agent 重塑。**ACP 作为
终端 ↔ agent 标准协议**的可能性变高——这是 LSP 范式（语言 ↔ 编辑器）
在 agent 时代的对应物。

**关注度**：⭐⭐⭐⭐ — 仅 2 条评论，PH 用户不是它的目标人群，但作为
"协议标准化早期信号"值得记录。
