### 1. Warp Open-Source · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/warp) | [GitHub](https://github.com/warpdotdev)

**定位**：商业 agentic 终端 / 开发环境 Warp 把核心仓库开源给社区，并
内置 "Oz-managed agents" 来做大头的 coding / planning / testing 工作，
社区贡献者主要负责"想法、方向、验收"。week one 25K+ star / 500+ contributor。

**差异点**：和"开源 IDE / 终端"这条传统线（VS Code / Zed / Ghostty）
最大的差异是 **贡献流程本身就是 agentic**——不是社区写 PR、维护者
review，而是 maintainer-managed agent 写 PR、社区做产品方向和验收。
更接近 Cursor / Cognition 这条线的做法但是反过来：闭源 agent + 开源
代码库 + agent 做大部分代码、人做大部分判断。Zed 是"开源代码，团队
全员写"，Warp 是"开源代码，agent 全员写"——两种 OSS 的劳动分工。

**技术判断**：Oz 是 Warp 内部多 agent 编排系统的代号（之前在他们 blog
里露过），从外部看就是"一组按 ticket / spec 跑的长任务 agent，统一
受人类维护者指令"。把它放到 OSS 项目上做大规模实验，意味着 Warp 觉得
自己的 agent 已经能在外部仓库（不只是 Warp 自家代码）独立产出可
merge 质量的 PR，这个 bar 比 Devin / Cursor agent 在内部代码库上的
demo 高一档——外部贡献者会用很挑剔的 review 标准检验。

**信号价值**：这是 agentic OSS 工作流第一个量级足够大的活体实验。如果
"agent 做 90% 代码 / 人做 10% 方向"在 Warp 自家项目上跑得通，下一步
就是这套范式被复制到别人项目，maintainer 的工作描述会被重新定义。
另外一个观察是 **Warp 选择把"贡献规则改造"当作开源的最大卖点**，
而不是任何技术创新——说明终端 + agent 这个垂直已经卷到"工作流" / 社区
机制是稀缺的差异化资源。对做开发者工具 / agent 框架的，值得跟一下
Oz 的 governance 模式，这可能比代码本身重要。

### 2. Weavable · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/weavable) | [Website](https://www.producthunt.com/r/33OUNMSCT7MK3F)

**定位**：通过单一 MCP endpoint 给 agent 提供"持久、实时的工作上下文"——
把公司现有工具里散落的更新、对象关系、系统变更聚合成一个 agent 可
查询的 context layer，让 agent 不必每次都重新 ingest 数据。

**差异点**：常见的"agent 记忆 / 知识库"产品（Mem0 / Letta / Zep）大多
做"对话 / 任务记忆"——存的是 agent 自己产生的状态。Weavable 反过来：
存的是**公司已有系统里发生的变化**（CRM 状态变更、ticket 流转、
文档更新），作为外部 context 喂给 agent。最接近的对位是 Glean
agent + RAG over enterprise data，但 Weavable 走 MCP 协议而不是
自有 API，且强调"事件流式"而不是"全量检索"——agent 不用每次都 RAG，
拿到的是已经按关系归类的最新 snapshot。

**技术判断**：单 MCP endpoint + "事件流式增量" 是关键工程取舍。这意味
着背后大概率是：（1）一组 connector 监听各家 SaaS 的 webhook /
changelog，（2）一个 entity graph 维护 cross-system 关系，（3）MCP
server 把这套 graph 投影成 agent 可调用的 tool。token 节省的说法
（"reason without re-ingesting"）只在 graph 投影做得好的情况下成立——
否则就退化成"普通 RAG over enterprise data"。值钱的是中间那个 entity
graph 的 schema 设计。

**信号价值**：MCP 已经从"协议宣传期"进入"被当作产品形态的卖点"阶段——
Weavable 整个 pitch 就是"single MCP endpoint"，不再需要解释什么是 MCP。
另外，"context layer as a service" 这个品类在 5-09 的 Agent-Ready Docs
Benchmark / Nylas CLI 已经露过苗头，今天 Weavable 把"持久企业上下文"
做成独立产品——这是 agent infra 分工进一步细化的信号。做 agent 框架
的可以考虑把 enterprise context 这块明确划出来交给上游，自家专心做
推理 / 规划。

### 3. Web Speed · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/web-speed) | [Website](https://www.producthunt.com/r/EORFJWPOR7CX6S)

**定位**：给 web agent 用的"逻辑层"——把任意网站翻译成 token-efficient
的"机器地图"（machine map），声称 agent 用它跑网页能省 70–90% token，
速度和可靠性都更高，关键是 mapping engine 是 **deterministic 的**。

**差异点**：和今天榜单上一堆 browser-use / computer-use agent
（Plouton AI 在 FinOps / Codex in Chrome 5-09 / Anthropic Computer Use）
最大的差异是它**不试图自己做 agent**——它做的是"网页 → 紧凑表示"
这一步的中间件。Browser Use 把整个 DOM dump 喂给模型，Web Speed 是
"先 deterministic 解析成结构化 map、再让 agent 在 map 上 reason"。
最近的对位是 Browserbase 的 stagehand（DOM-aware action layer）和
ScrapeGraphAI（DOM → graph），但这两个都还是 model-in-the-loop，
Web Speed 强调 "deterministic mapping engine" 是没有模型参与的——
更像是 lex/parser 给 agent 看。

**技术判断**：deterministic mapping 大概率是基于 CSS selector / DOM
heuristics + 已知模板库的混合方案——这种"先 deterministic 压缩、
再交给 LLM"在 Stagehand / Sequoia 的几个相关投资里都见过类似思路，
但单独把它做成 standalone 产品（而不是 framework 的一部分）是新的。
90% token saving 这个数字大概率指的是"vs 直接 dump 整个 rendered DOM"，
比 baseline 强，但 vs 一个写得好的 selector-based scraper 不一定有
那么大优势。

**信号价值**：web agent 这条线正在分化出"模型不动 / 中间件优化"的细分
赛道。"agent 太贵 / 太慢"的根本问题大家都意识到了，方向有三条：
模型变便宜（Haiku 4.5 / DeepSeek）、推理走 cache（prompt caching /
prefix cache）、**输入端做压缩**（Web Speed / 各种 token-efficient
代理）。第三条之前主要是论文 / 学术 prototype，今天是它进入商业
产品的早期信号。对做 agent SDK 的，"web 输入的 token efficiency"
可以作为下一个 plug-in 形态的入口去看。

### 4. Suprbox · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/suprbox) | [Website](https://www.producthunt.com/r/IGTQIGCZCG2J7H)

**定位**：给 AI agent 用的"策略门控"数据保险库。不要再把 Drive / S3
key 直接交给 agent；改成给一把 scoped Suprbox key，每次读取都按规则
（敏感度 / 时间 / 频率 / 人工审批）校验，并写不可篡改的 audit log。
口号是 "protect the data itself, not the prompt"。

**差异点**：现在 "AI agent 安全"这块挤了三类玩家：（1）prompt
guardrails（NeMo Guardrails / Lakera），管模型输出；（2）runtime
sandboxing（Modal / E2B），管执行环境；（3）secrets vault for agents
（Doppler / 1Password 也在做）。Suprbox 是 **(2)+(3) 中间的一个新
切片**——它不管执行环境也不管 secret 本身，管的是"agent 拿到 key 之后
怎么读 data"。最近的对位是 Cloudflare 的 AI Gateway（egress 网关）
和 OPA / Cedar 这类策略引擎，但 Suprbox 把它**专门收敛到"agent 读
文件 / 对象存储"这个动作上**，做了一套自有 policy DSL 和 sub-200ms
verification API。今天榜上还有 ClawSecure（pre-install 扫描 + runtime
monitoring）走的是病毒查杀类比，Suprbox 走的是 IAM 类比——后者商业
模型更清晰。

**技术判断**：核心是一个 policy engine + key proxy + immutable log，
都是成熟工程组件。难点不在代码而在 **policy DSL 的表达力 vs 简洁度**
平衡——OPA / Cedar 都死在"太通用 → 难写策略"上。"sensitivity /
time / rate / human-approval" 这四个维度看起来是刻意收敛过的，
就是个 mini-policy 而不是图灵完备 DSL，这是对的选择。Audit log
immutability 大概率是 append-only + Merkle root，没什么新东西。

**信号价值**：今天 Suprbox + ClawSecure + Known Agents 同日 launch
是一个明显信号——**"防 agent" 类产品从"被讨论"进入"被买单"阶段**。
之前 agent security 多半是甲方 internal tooling，第三方很少敢卖，
原因是企业不愿意把 agent 行为暴露给第三方。今天连续三家把它做成
公开产品并强调"open / free to start"，说明（1）市场认知到了，
（2）部署形态找到了不需要触达内部数据的切角（policy proxy 在外
而不是 agent runtime 在内）。对做 agent framework 的，"data
boundary"这一层可以开始当作上游来用而不是自己做。

### 5. Known Agents · ⭐⭐⭐
[PH](https://www.producthunt.com/products/known-agents) | [Website](https://www.producthunt.com/r/K56IE2TRUY7CJD)

**定位**："给 bot 用的 Google Analytics"——实时监测访问你网站的 AI agent /
crawler / scraper：它们是谁、看了哪些页面、从哪来、把哪些人类用户
导给你。

**差异点**：传统 bot analytics（Cloudflare bot management / DataDome）
出发点是**防御**——区分好 bot / 坏 bot 然后阻断坏的。Known Agents 出
发点是**理解**——AI agent 已经是流量的一半且会买你产品，所以你需要
像看人类用户那样看它们的旅程。最近的对位有 Vercel 的 AI Crawler
Stats（很基础）和 humans.txt / llms.txt 这条标准化路线，但 Known
Agents 是第一个把 "agent traffic analytics" 做成独立产品并强调
"agent referral to humans"（也就是 agent 看完你的 doc 推荐人类来）。

**技术判断**：技术深度有限——主要是 user-agent / IP range 维护 +
behavior fingerprinting + dashboard。值钱的是它的 **labelled agent
identity 数据库**（哪些 UA 是 ChatGPT / Perplexity / Claude / 自部署
agent / 恶意 scraper），这块需要长期维护、很难一开始就好。一个
风险点是 anti-fingerprint 大势之下，agent 主动伪装的比例可能很快
攀升，让这种"按身份归类"的产品越来越像"按 IP 归类"。

**信号价值**：这个产品反映的潜在叙事比产品本身更重要——**网络流量
组成正在从"人为主、bot 是噪声"转向"agent 是 first-class 用户"**。
如果"AI agent 是流量一半"这个数字（虽然没具体来源）即便夸张一倍
也是真的，那 SEO / 内容策略 / 转化漏斗这套老活就要重写。短期看
Known Agents 自己面临"客户愿不愿意为 bot 行为分析单独付费"的问题，
但中期看"为 agent 优化网站"这个咨询 / 工具市场会被打开。
