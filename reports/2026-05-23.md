### 1. Google Antigravity CLI · ⭐⭐⭐⭐⭐
<https://www.producthunt.com/products/google-antigravity> | <https://www.producthunt.com/r/2VVOKDD4COMAK2> · 239 votes / 20 comments

**定位**：Google 把 Antigravity（11 月发布的 agentic IDE）的 CLI 形态
搬上来——multi-step reasoning、multi-file edit、tool calling、persistent
history，专门为 SSH 会话和键盘流优化。

**差异点**：直接对位 Claude Code（Anthropic）、Codex CLI（OpenAI）、
Aider、Cursor CLI。这一年 IDE 端 Cursor / Antigravity 拼得激烈，但
**SSH / 键盘流**这一段一直是 Claude Code 一家独大；Google 这次主动
把战场拉到终端，是承认"开发者真正长时间停留的环境是 shell 不是 IDE"。
另一层差异是模型——内嵌 Gemini 3 Pro 的 1M context 在终端里第一次
变成可被 piping 的资源。

**技术判断**：从描述看是和 IDE 同源的 agent runtime，只换了前端表层
（terminal UI + readline）。"persistent history"那一句暗示已经把
trajectory / state 拆成可序列化的格式，否则在 SSH 里 resume 不了。
真正考验是 tool calling 的本地 fs / shell 权限模型——Claude Code
靠的是 `--allowedTools` 白名单，Antigravity CLI 没说，但既然是 Google
出品，大概率会接 Google Cloud 的 IAM / OAuth 套路，对个人开发者可能
反而是阻力。

**信号价值**：CLI agent 三足鼎立的局面正式成形（Anthropic / OpenAI /
Google）。对做自家 agent 的同学，**协议层（trajectory format、tool
calling schema）很可能在未来 3–6 个月被强制收敛**——值得现在就开始
关注 Antigravity CLI 输出的 transcript 结构和它是不是兼容 OpenAI 的
Responses API。同时也说明：自己再做"一个更好的 Claude Code 平替"
窗口期基本关闭，做 **在 CLI agent 之上的工具链**（MCP 服务、observability、
权限层）才是缝隙。

### 2. Memdex · ⭐⭐⭐⭐
<https://www.producthunt.com/products/memdex> | <https://www.producthunt.com/r/LBMOK2KFA74TNE> · 244 votes / 28 comments · c/v=0.115

**定位**：Chrome 扩展，自动捕获 ChatGPT / Claude / Gemini 三家 web 端
的对话，加密存到本地 IndexedDB，下次写 prompt 时像 Grammarly 划下划线
那样高亮"过去聊过的可复用上下文"，一键注入。

**差异点**：这条赛道已经有不少玩家——OpenAI 自己的 Memory、
Letta（前 MemGPT）、mem0、Cognitive Memory 等。Memdex 的差异是 **三家
通吃 + 本地 + 浏览器原生交互**：不要求改 LLM 端、不要求 API key、不上传
任何对话到云。Grammarly 式的"下划线提示再注入"是真把 memory 当输入法
辅助来做，比 mem0 那种 SDK 接入侵入度小一个数量级。

**技术判断**：Chrome extension + IndexedDB + 加密——本地 vector store
基本是必选项（要做语义匹配才能"识别可复用上下文"）。难点在两块：(a) 三家
DOM 改一次就要适配一次，工程量持续累积；(b) "Grammarly 式注入"要在用户
还没提交时实时分类，得有个轻量本地模型或者远端 inference。Pro 版本可能
卡在这一层。

**信号价值**：跨 LLM 的 memory 层在过去半年从概念变成了真有人愿意付钱的
产品。对做 agent 的同学：**memory 不应该绑定到某一家 LLM 服务**——这条
原则今天有了用户验证。如果做 Claude Code 自己的 routine，可以学这个
Grammarly 式 UX：让 memory 推荐自己冒出来，而不是用户主动 `/recall`。

### 3. Vibedock · ⭐⭐⭐⭐⭐
<https://www.producthunt.com/products/vibedock> | <https://www.producthunt.com/r/I7KXQ2JJX74MTI> · 95 votes / 5 comments

**定位**：macOS 菜单栏小工具，一键开关已配置的 Claude Code MCP 服务器，
切换后自动 kill + restart Claude session 让改动生效。

**差异点**：这是**今天最对我胃口的产品**。问题描述精准到不能再精准：
"每个 MCP 服务器都会把它的 tool definitions 塞进每个 message 的 context
window，不管你这次会不会用它"——这就是 Claude Code 用户每天都要踩
的坑（MCP 一多，上下文先被 tool schema 吃掉一截）。Anthropic 自己
还没出官方解决方案，目前的 workaround 是手动改 `~/.claude.json`。
Vibedock 把这件事变成菜单栏点一下。

**技术判断**：核心就两步——读写 `~/.claude.json`、`pkill` 当前 Claude
进程再让用户重启。听起来 trivial，但 "自动 kill 后让 session 不丢"
那一段隐含了对 Claude Code session resume 协议的反向工程，工程量被
低估了。值得注意的是它现在只覆盖 macOS，Linux / WSL 上同样痛但没人做。

**信号价值**：**Claude Code / MCP 周边的"原生工具链"开始浮现**——
这是平台演化到一定阶段才会出现的信号。和昨日 DCP（agent 权限层）
是同一类：不是再做一个 agent，而是把 agent runtime 周围的能力一件件
拆出来做产品。如果做 Claude Code 配套工具：**MCP toggle、context
inspector、conversation diff、cost dashboard** 这四个方向都还没被
认真做。Vibedock 占了第一格。

### 4. Forsy · ⭐⭐⭐⭐
<https://www.producthunt.com/products/forsy> | <https://www.producthunt.com/r/A2U7PH6SGMMNIE> · 114 votes / 5 comments

**定位**：从你已经在用的 agent（Claude、Codex、OpenClaw、Hermes）抓
workflow 数据（trajectory），打包成结构化、可授权的数据集卖出去，定位
是"agent workflow 数据市场"。

**差异点**：数据市场本身不是新概念（Scale、Surge、Snorkel），但他们都
是"雇人标"的。Forsy 走的是**被动 telemetry → 结构化 → 市场**这条路：
让真实使用中的 agent 轨迹变成可被 RL 训练用的数据。这个思路最像
Common Crawl 之于 LLM 预训练——把已经在发生的事情捞起来卖。

**技术判断**：技术难点不在抓取（Anthropic / OpenAI 的 SDK 都有 hooks），
而在两端：(a) 谁愿意把自己 agent 的 trajectory 卖出去——必须做匿名化
和敏感数据剥离，但 trajectory 里几乎一定带凭据片段或代码 IP；(b) 谁
愿意买——目前 agent training 数据的有效买家就是 Anthropic / OpenAI /
Google / Meta，他们更可能自己采购或自家用户共享，不太需要中间商。
这两端能不能跑通，决定了 Forsy 是真生意还是 thesis。

**信号价值**：**Agent trajectory 作为新型训练数据这件事第一次被正经
产品化**。之前 RL on real agent traces 还停留在论文阶段（Reflexion、
ReST、SWE-RL）。即使 Forsy 自己跑不出来，"trajectory 是新石油"
这个 thesis 值得记住——以后看到任何 trajectory 收集 / 标注产品都
该回到这条线去对位。

OpenAI / Anthropic API，明确把 Cohere / Llama / Qwen 作为 backbone
candidate 加进去是越来越必要的。
