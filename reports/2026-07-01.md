### 1. Tabstack Browser Automation · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/tabstack) | Built by Mozilla

**定位**：一次 API 调用就让 agent 在真实浏览器里完成多步任务——导航、
点击、填表、等待交互后才渲染的页面——浏览器和 LLM 都由服务端托管，
调用方不用自己维护 Chromium 集群。

**差异点**：和 Browserbase / Anchor / Steel 等一批 "agent 用浏览器"
基建同赛道，但 Tabstack 明确走 **accessibility tree** 路线而不是
screenshot + vision，官方数据宣称 token 消耗低 60–80%。定位是"API
而不是 framework"——不需要在应用侧装 SDK 组装 loop。Ephemeral 会话、
不用数据训练，卖点更偏企业。

**技术判断**：a11y-tree 路线是有工程判断的选择——vision 路线对小组件、
密集表单、长表格都不友好，而且 token 成本随分辨率爆炸；a11y-tree 直接
拿到语义结构，在文本 LLM 里推理更省。由 Mozilla 出品意味着 Gecko 的
可控性可以直接吃到（DOM/AXTree 无缝拿）。声称"no concurrency ceiling"
背后是 Firefox headless + tab pool 的水平扩展。

**信号价值**：agent 浏览器基建赛道从今年上半年就在挤，Mozilla 这一
下场把竞争抬到基础平台维度。对我关心的方向——**做 agent 产品的团队
不应该再自己养 Playwright 集群**——今天算是又一个证据点。

### 2. Sequence Agentic · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/sequence) | 237 votes · 98 comments

**定位**：给 AI agent 做"钱的执行层"——发起转账、拆分、按规则路由，
覆盖银行账户、卡、SaaS 账单、贷款。Scoped API key + 服务端花销限额 +
全量 audit log，agent 不持有用户凭证。

**技术判断**：区别于 Plaid/Increase 这类只读或单一 rail 的 fintech
API：Sequence 直接暴露"移动资金"的写路径，并把风控（花销上限、审计）
下沉到服务器，让 agent 端只拿受限 key。称已在合规 rail 上处理 $3B——
如果属实，这不是 demo 阶段的产品，而是把已有的 fintech 中台包了一层
agent-friendly API 对外。

**差异点**：和 Skyfire、Nexcap、Payman 一批 "agent 支付" 同赛道，但
Sequence 把 Claude / n8n / Zapier 明确列为一键集成对象，说明它把
distribution 押在既有 agent stack 上而不是自建 SDK。

**信号价值**：agent 经济里"付钱这一步"过去一年一直是空白——大家都在
给 agent 加检索、加浏览器、加代码沙箱，但真让它 spend money 的产品
基本没有。Sequence 是这条线里比较硬的一枪。给做 agent 的团队一个
现实提示：**支付、KYC、audit 这些合规栈不用自己搭，可以直接买**。

### 3. N71 · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/n71) | 140 votes · 38 comments

**定位**：给多 agent workflow 建一个共享的、随时更新的上下文——一张
"活的知识图"，通过 MCP 暴露给你正在用的一堆 agent，任何 agent 都
"已经读过"其他 agent 刚刚讨论出来的结论。

**差异点**：常见的做法是 pgvector + RAG，每个 agent 各自查各自的
memory；N71 走 graph + MCP 路线，语义单位是"事实节点"而不是
"chunk"，更新是事件驱动的（工具连上以后新事件自动进图）。对标物是
Mem0、Zep、LlamaIndex Agent Memory，但 N71 明确定位是"跨 agent 共享"，
而不是"单 agent 长记忆"。

**技术判断**：living knowledge graph + MCP 是个正确的组合——MCP 天然
适合把只读 context 暴露给若干消费方 agent，把写入路径限制在少数几个
可信 agent，避免 memory 污染。挑战在 conflict resolution：当两个 agent
给同一 entity 写入相反事实时怎么办，从 launch page 看不出细节。

**信号价值**：过去半年主流叙事是"单个更强的 agent"，但真正的生产
场景越来越多是"一群窄 agent 协作"。共享 context 是这条路的必配件。
N71 的产品定位精准踩在这个空缺上。

### 4. Adam CAD Copilot · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/adam-cad-copilot) | 310 votes · 77 comments

**定位**：在 Onshape 和 Autodesk Fusion **内部**跑的 AI CAD 助手——
prompt 建/改零件、引用已选几何、清理 feature tree，全部保留可编辑
性（parametric，不是死几何）。

**差异点**：区别于 Zoo.dev / CADdy / Spline 那批"新一代 AI-native CAD
编辑器"路线——Adam 反其道而行，插进 mechanical engineer 已经在用的
Onshape/Fusion 里。这个选择意味着放弃"完全掌控 UX"，换来"用户不用
迁移工作流"。对早期 AI-in-vertical 产品，这一般是更容易过 PMF 的路径。

**技术判断**：难点在保持 feature tree 的干净——LLM 生成 parametric
CAD 操作而不是 mesh 输出，需要在 Onshape/Fusion 的 API 上做符号化
建模，而两家 API 都不算轻量。"reference selected geometry"说明它
读用户 selection，能做上下文相关的操作，这是把 copilot 做进真实工作
流的最小要求。

**信号价值**：AI 进入垂直专业工具（CAD、DAW、EDA、GIS）今年会陆续
爆发。Adam 是一个执行度看上去比较踏实的样本：不做全新编辑器，做
现有工具的插件。

### 5. Acti · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/acti-2) | 548 votes · 231 comments · comment/vote 0.42（今日最高）

**定位**：iOS 键盘，长按 Acti Bar，一句自然语言拿到结果或直接触发
动作——不切换 app，比赛比分、附近餐厅、Notion 文档、LinkedIn、Meet
链接、Calendar 都在原地返回。

**差异点**：不是又一个 chatbot，也不是又一个 launcher（Superhuman
Command / Raycast Mobile 那种）——它把入口塞在**键盘层**。这意味着
在任何输入框都触发，不需要用户"退出去问"。相比之下 Arc Search 是
浏览器起手，Perplexity iOS 是独立 app。

**技术判断**：iOS 自定义键盘做代理入口过去一直被 App Extension 内存
限制卡住，做重的意图理解基本不可能；从描述看 Acti 是键盘本地做
UI + 云端做 intent 解析 + tool 调用。要好用，核心是"我按下之前 app
已经知道我大概想干嘛"——他们 231 条评论的高互动率说明这一段做得
不算差。

**信号价值**：agent 的入口今年还没有稳定答案——桌面上有 Raycast /
Claude Desktop，浏览器有 Arc / Perplexity，移动端一直空缺。Acti 押
"键盘"这个位置比押 "app icon" 更接近 iOS 的操作系统层。方向可以
关注。
