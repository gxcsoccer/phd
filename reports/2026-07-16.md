### 1. Codex Micro · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/openai) | [Website](https://www.producthunt.com/r/N2GRCRNINJWBZ7)

**定位**：OpenAI × Work Louder 联名的实体控制器，为 Codex 的编码 agent
做"物理外设"——键盘上有专门的 skill 快捷键、一个用于调 reasoning level
的旋钮，以及能实时反馈 agent 状态的 RGB 灯带。

**差异点**：这是本轮 AI 硬件里第一个把"agent 状态 / 推理档位"作为一等
输入面的东西。Rabbit R1 / Humane Pin 都还在"AI 助手 = 手机替代品"这条
错误路线上；Codex Micro 直面一个真实存在的问题——LLM agent 长时间跑
后台任务时，人类需要一个不占屏的仪表盘和一个不打断上下文的干预入口。
Warp block / Raycast 的 hotkey 只解了后一半，没解仪表盘的一半。

**技术判断**：硬件侧是 Work Louder（他们此前做过 Loupedeck 风格的模块化
键盘），非常规工艺量级，说明 OpenAI 是"生态推动"而非"造一台设备"，
更接近 Anthropic 出 SDK 的调性。功能面上"reasoning dial"暴露了一个
Codex API 端能吃到的参数——这暗示 Codex agent 已经把 reasoning level
真正拆成了可运行时切换的 knob，不再是模型固定档位。

**信号价值**：AI-native 外设的定义可能正在从"AI 音箱 / AI 眼镜"这条
终端路线，回到"给已经天天用 AI 的开发者做的键盘、macro pad、状态
面板"这条更朴素的路线。做 dev tools / agent 平台的团队值得盯：
一旦有可控的"agent 状态总线"，物理面板、菜单栏 widget、smartwatch
complication 都会随之长出来。

**关注度**：⭐5——OpenAI 本身的曝光量 + 全新形态 + 对"agent runtime 可
观测性"这条工程线的暗示。

### 2. Zro · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/zro) | [Website](https://www.producthunt.com/r/VLHFXUWJ6KN7JY)

**定位**：为 coding agent 打造的推理后端。开源模型、多区域部署、
零请求留存（zero request retention），主打隐私与延迟。

**差异点**：市面上做"cheap LLM API"的一堆（Together / Fireworks /
DeepInfra），但真正把"coding agent"当作产品定位的很少——Zro 直接
把 Aider / Cursor / Claude Code 这批客户拎出来。零留存这一点很硬：
Together 等大部分推理服务默认会做请求日志（哪怕短期），
对企业代码库来说是硬门槛。今天多数 Fortune 500 拒绝把内部代码接
公有 LLM，多半就是这一层合规过不去。

**技术判断**：多区域 + 零留存这套组合，说明底层是每区域独立的
GPU 集群 + 无中央 log store。对 coding agent 优化过意味着大概率
支持流式 diff、长上下文（>200k）、结构化输出。开源模型侧押注
Qwen 3 Coder / DeepSeek V4 一类，避开与 GPT-5.x / Sonnet 5 的
正面竞争而卡"可自证不留数据"的差异位。

**信号价值**：AI 基础设施的分层正在往"数据主权"方向漂——2024
的关键词是"便宜"，2026 的关键词开始变成"我能证明你没存"。
做 agent 平台的产品要么自建后端，要么就等"privacy-first 推理"
成为可选项，然后接进来。这对做 local-first / on-device 的团队
是好信号：中间态（远程推理 + 强合规保证）会先跑起来，纯本地
是下一步。

**关注度**：⭐5——命题击中 dev tools 生态最痛的合规缺口，Vercel
Day 榜二名不是偶然。

### 3. Cito · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/cito) | [Website](https://www.producthunt.com/r/LY3LQ5RV2PFQTT)

**定位**：为 agent 打造的学术搜索引擎。底库 Semantic Scholar 236M
论文（其中 146M 有 SPECTER2 dense vector），倒排 + 稠密 RRF 融合 +
cross-encoder reranker，暴露 JSON API 和原生 MCP 端点。

**差异点**：作者原话是"因为学术 API 全都把我的 agent throttle 到死才
做的"——这是一个纯粹为 agent workload 设计的 retrieval 服务。市面
上给人用的学术搜索（Semantic Scholar / Consensus / Elicit）都不假设
"一个 agent 会一小时打几千次 query"，速率限制、身份认证、返回结构都
不吐给 agent 用。Cito 直接跳过 UI 层，从 MCP endpoint 起步。

**技术判断**：SPECTER2 是 AllenAI 面向科学文本的 embedding 模型，用
它是懂行的选择；RRF（Reciprocal Rank Fusion）+ cross-encoder rerank
是当下 hybrid retrieval 的教科书配方。真做出来意味着有一整套向量
索引运维（146M×768 dim 向量约 450GB，加倒排索引另外几十 GB），不是
套壳。

**信号价值**：agent-native retrieval 是接下来一年的高增长赛道。
"给人用的搜索"和"给 agent 用的搜索"在延迟、并发、结构化程度、
认证形态上都不一样。Perplexity / Exa 之外，垂直领域会长出各种
MCP-first 的检索层。Claude Code 用户可以直接 hook 一个。

**关注度**：⭐4——票数不高但工程含量顶尖，且"MCP-first retrieval"
这条线未来一年会大爆。

### 4. Nitrosend · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/nitrosend) | [Website](https://www.producthunt.com/r/EJGO5YGEWQK2VN)

**定位**：为 AI agent 设计的完整 email 栈。文案里最有意思的一句：
"Point any agent at nitrosend.com/SKILL.md and it signs itself up,
onboards, connects your domain and sends."

**差异点**：Resend / Postmark / SendGrid 都是给"人工程师"用的
transactional email；Nitrosend 把入口翻过来——SKILL.md 是给
Claude/GPT 一次读完就能完成注册、绑定域名、发出邮件的"agent
onboarding script"。这是我第一次看到有产品把 SKILL.md（Claude
Code 的 skill 元描述格式）作为公开外部接口暴露。等价于把"agent
读一次文档就能自主接入 SaaS"当作 SDK 契约。

**技术判断**：底层大概率是 Resend / Postmark 上包装（或者自建 SMTP
+ SES），价值在语义层：把 SaaS 的注册流转成 agent-consumable
的 markdown 指令序列。Email inbox for agents（agent 有自己
domain 的邮箱、能收人回复）这一块比"AI 发邮件"更有意思——真正
让 agent 参与 email 对话，而不只是单向 outbound。

**信号价值**："SKILL.md as public API"是一个可以广泛复制的模式。
所有面向开发者的 SaaS 都可以给自己写一份 SKILL.md，agent 一 curl
就能完成 100% 无 UI 的接入。是继 OpenAPI/GraphQL/MCP 之后的第四
波"machine-consumable API"浪潮，且门槛更低——直接 markdown。

**关注度**：⭐4——命题很小巧，但代表了 agent-native SaaS 的
一种成熟形态。

### 5. Manta AI · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/manta-ai) | [Website](https://www.producthunt.com/r/VROEAEJHEWF5ND)

**定位**：自主 Web 应用测试 agent——给 URL，它像真实用户一样探索
应用，map 出所有 flow，写自愈（self-healing）测试用例。可本地
运行，能测 firewall 后 / localhost 上的应用。

**差异点**：QA/E2E 测试这条线上早有 Playwright + LLM 混搭产品
（QA Wolf、Rainforest、Momentic），但 Manta 有两点跟大多数不同：
① agent 主动"探索"而不是"复现固定 flow"，减掉了"人先写脚本"
这一步；② 明确支持 self-host / 本地运行，可以打 localhost。
第二点特别重要——多数 AI 测试 SaaS 需要把内部环境暴露到公网，
企业根本没法用。

**技术判断**：主动探索型 QA agent 是 browser-use / Steel 一系
落地形态之一。"self-healing test case"通常等价于用 LLM 从最新
DOM + 历史 selector 里推断新 selector，同时保留 flow 的语义
描述。真做稳定不是 LLM 的问题，是 test flake 的经典问题：
需要一层可回放的 trace + 语义 diff。团队敢发就说明这层做出来了。

**信号价值**：QA 是 agent 用户价值最容易被验证的场景之一——
它输出确定性的 pass/fail，人类可以直接判断有没有价值。相比"帮
你写代码"要靠品味打分，"帮你测代码"能自证。做 dev tools 的
团队值得关注：这条线越成熟，"自动生成 + 自动验证 + 自动修复"
的三段闭环越近。

**关注度**：⭐4——不是最颠覆的一个，但是 agent + dev tools
里胜率最高的赛道之一，且本地部署这一点让企业采用门槛低。
