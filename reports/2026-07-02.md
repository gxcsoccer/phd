产品的都会遇到"我给 agent 塞什么 context 才够"这个问题，谁能把这一
层做成 API 谁就吃这波红利。

### 2. scritty · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/scritty) | [Site](https://www.producthunt.com/r/BOSLVJYFBR5RDQ)
124 votes · 50 comments · cvr 0.40（今日高互动榜第二）

**定位**：一个终端模拟器，把你所有 CLI agent（Claude、Codex、Copilot、
Antigravity、Ollama）的对话抓下来、索引成一个可搜的 corpus、再通过
MCP 反向暴露给这些 agent。桌面 / 浏览器 / 手机三端同步，抓下来的数据
留在本地。

**差异点**：这个产品的问题定义就是差异点——"跨 CLI agent 共享的
long-term memory 应该在哪里"。目前市面上的答案是：Claude 有
`~/.claude/history`、Codex 有 session 文件、Cursor 有自己的 chat log，
互不相通。scritty 的做法是"拦截在 terminal 这一层"，抓到的是所有 CLI
agent 都会走的 stdout/stdin，天然跨供应商。类似定位的 Milestones、
Polygraph 是从 IDE 或 project 层做，scritty 更底层。

**技术判断**：核心是一个 terminal emulator + 一个本地索引（大概率
sqlite + fts 或轻量向量库）+ MCP server 暴露 search 接口。cvr 0.40
在这个体量下是很高的互动率，说明 CLI agent 用户社群对"memory
fragmentation"的痛感是真的——不是 vendor 造出来的问题。本地优先
（"captures stay on your machine"）也踩中隐私诉求。

**信号价值**：印证了近两周的一个假设——**跨 agent 的横切基础设施
正在独立成品**。过去 memory 是每家 agent 自己的 feature；现在它变成
了独立产品，且是终端级别的横切。谁做这一层，谁就是 agent 时代的
Alfred / Raycast——用户人格化的入口。

### 4. Retrace · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/retrace-2) | [Site](https://www.producthunt.com/r/IXSYWU2YQILNKM)
100 votes · 33 comments · cvr 0.33

**定位**：给 AI agent 做时间旅行式 debug——录制 agent 每一步（LLM
调用、tool invocation、错误），可以回放、fork 到任意步、分享给同事。
1000 traces/mo 免费。

**差异点**：类似 Langsmith、Langfuse、Braintrust 已经在做 agent
observability，但通常是"事后看日志"。Retrace 卖点是 "**fork run**"
——在一次 agent 执行到某一步的中间态直接分叉出去，改 prompt / 换
tool / 换 LLM 再跑下半段。这是从 CI 时代的"重跑失败的 job"进化到
LLM 时代的"改变量重跑执行"，概念更接近 rr、Domain Reload、或者
notebook 的可编辑历史。

**技术判断**：fork 能力要求把 agent 的每一步执行都存成可 replay 的
snapshot（工具调用参数、返回值、LLM messages、prompt cache 状态），
且要 patch 掉外部 side effect（数据库写、API 调用）。这是很深的
runtime 工作，不是套一层 tracing SDK 就能做。cvr 0.33 说明做 agent
的人立刻就理解这个价值——他们每天都在"想改一步再重跑，但要从头
跑很贵"。

**信号价值**：AI agent 开发工具链的成熟度在补齐"debugging"这一
环。过去两个月 tracing 层（Langfuse、Traceloop）已经拉出来，
Retrace 是往上一层的 fork/replay。**下一步会出现"AI-native TDD"**
——把"某个 agent run 走到某一步应该出现某种 tool call"变成断言。
