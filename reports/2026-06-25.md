### 1. Heron · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/heron) · open-source · 105 votes · 21 comments

**定位**：用 eBPF 在内核层旁路抓 AI agent 的网络流量，把哪条进程发了哪次 LLM 调用、调用了什么模型、参数是什么全部还原出来——本质是给 agent 跑的机器装一个被动 sniffer。

**差异点**：现有 agent 可观测方案（LangSmith、Helicone、OpenLLMetry、Arize）几乎全是 SDK / proxy 路线，要改代码或者强制走中转 endpoint。Heron 完全不接触应用层：eBPF hook 在 TLS 之外读，所以即使 agent 用各自不同语言、SDK 没埋点，甚至直接 `curl https://api.anthropic.com/v1/messages` 走原生 HTTPS，都能被还原成结构化事件。这是过去这一年 LLM observability 赛道里没人占住的位置。

**技术判断**：eBPF + TLS 解密通常意味着 SSL_read/SSL_write 的 uprobes（OpenSSL/BoringSSL/NSS 分别 hook 一次），落到 ring buffer 后用户态 reassemble HTTP/2 流。难点不在原型，难在版本适配（OpenSSL 1.1 vs 3.x、Go 静态链接、Node.js 的 BoringSSL fork）。能在 PH 上抛出来说明至少在主流栈里跑通了，是真做工程的团队，不是套壳。

**信号价值**：当公司里跑的 agent 越来越多、来源越来越杂（自研 + Cursor + Claude Code + 一堆 MCP 黑盒），SDK 路线注定覆盖不全。"Wireshark for agents" 把可观测性从 instrumentation 模式拉回到 monitoring 模式，是 agent infra 成熟到一定规模后必然出现的形态。值得跟踪——尤其想做 agent 安全 / 合规审计的可以借鉴。

### 2. BrowserAct · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/browseract) · 532 votes · 105 comments

**定位**：给 AI agent 的远程浏览器 runtime，主打绕过反爬封禁、登录态保持、批量并行任务、表单/上传/验证码这些 headless 浏览器在生产里最难搞的边缘场景。

**差异点**：和 Browserbase、Hyperbrowser、Steel.dev、Anchor Browser 站在同一赛道。BrowserAct 的描述把"pass blocked pages、handle verification"放在最前面，强调 evasion 和 session 持久化——这是 Browserbase 一直被吐槽的痛点。但描述里看不到 CDP / Playwright 兼容性细节，也没有定价/并发的硬指标。

**技术判断**：从 tagline 看产品形态比较成熟（指纹池 + 代理池 + verification handling），但描述里"clean web data for reasoning"这种话术更像 selling，不是炫技。属于在已经被验证的产品形态上做"更好的版本"，工程门槛中等偏上，但创新度低。

**信号价值**：browser-for-agents 这一层已经卷成红海，今天榜单上还有 BrowserBash 同时出现（CLI + 本地 Ollama 跑的开源版）。值得关注的是这条赛道开始往两端分化——上面是 BrowserAct 这种托管 runtime，下面是 BrowserBash 这种 local-first CLI；中间的 Browserbase 类 SDK 反而被两头挤。

### 3. Polygraph · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/polygraph) · 154 votes · 25 comments

**定位**：给 AI coding agent 做"跨仓库的依赖图 + 跨 session 的记忆层"——把你公司所有 private / public repo 拼成一张统一的依赖图，让 agent 知道"我在 service-A 改了一个接口会影响哪些 caller"，并且让一个 agent session 的工作可以被另一个开发者、另一台机器、甚至另一种 agent 接力。

**差异点**：和 Sourcegraph Cody Enterprise、Greptile、Cline 的 memory bank、Cursor 的 multi-root workspace 都重叠，但定位更"meta-harness"：不绑定具体 agent。如果实现到位，本质是 agent 的"git remote"——任何 agent 都能 push/pull session 状态。这个抽象层之前几乎没人单独做。

**技术判断**：技术难点在两块——(a) 依赖图：要 parse 多语言、识别 monorepo 边界、handle 私有 registry；(b) session memory 跨 agent 兼容：要么是一个开放 schema（类似 Memory MCP），要么得给每个主流 agent 适配。从 tagline 看更像前者。如果真做成 schema-level 的标准，是潜在的网络效应产品。

**信号价值**：coding agent 这一波的下一个真问题就是"如何让多个 agent 协作 + 上下文持久化"——单 session 内的能力（Claude Code、Codex CLI、Cursor agent mode）已经卷得差不多了。Polygraph 站对了位置。即便他们自己做不大，这个方向值得跟。

### 4. Figma Motion · ⭐⭐⭐
[PH](https://www.producthunt.com/products/figma) · 287 votes · 10 comments

**定位**：Figma 在主画布上加了一个真正的时间轴动画系统，原地复用现有 component / variable / auto-layout，不需要导出到 Rive / After Effects / Lottie。

**差异点**：Figma 之前的 Smart Animate 是 frame-to-frame 插值，玩不出复杂的 motion。市面上独立工具（Rive、Jitter、Cavalry）做得很好但属于"另一个软件"。Figma Motion 的核心赌注是"上下文为王"——动画师不用切换工具、不用重建组件，这件事真做对了能直接干掉一批 SaaS。

**技术判断**：动画引擎不难，难的是把现有 Figma 的 reactive 数据模型扩展到时间维度还不破坏现有文件格式。10 个评论 / 287 票，评论率 0.035 异常低，说明这是 Figma 官方 broadcast 来的流量，而不是工具圈在讨论——意味着大部分动画师还没真用上，要看实际能力。

**信号价值**：跟 AI agent 没关系，但这是这周唯一一个"老牌设计工具吞掉独立工具品类"的明确信号。Adobe 的位置会被持续蚕食。对做 design-adjacent tooling 的人，得想清楚自己的 differentiation 是不是终将被平台吃掉。

**定位**：Mac 原生的轻量项目计划工具（不是另一个 web SaaS），iCloud 同步，关键卖点是自带 MCP server——任何 agent 都可以读写它的 milestones / tasks。

**差异点**：项目管理类应用本来卷成血海。Milestones 的两点不同：(a) native Mac + iCloud-only 是隐私 / local-first 的明确站队；(b) MCP server 不是 bolted-on，是产品的核心卖点之一。这种"native app 把自己暴露给 LLM"的姿态在今天还很少见。

**技术判断**：MCP server 的实现量并不大，但选择把它当作主卖点说明产品哲学清晰——他们不打算去做 Notion AI / Linear Agents 那种"应用内 AI 助手"路线，而是相信用户会从 Claude / Cursor / 自家 agent 来操作 PM 工具。这是更分布式的赌注。

**信号价值**：这是今天榜单上 MCP 出现的第三次（Brain² by ClickUp、Papermark Agents、Milestones），其中 Milestones 是最纯粹的"我就是为了让 agent 用我"。看起来 2026 下半年，MCP server 正在从"加分项"变成 productivity SaaS 的标配——不带 MCP 的 PM / 笔记 / CRM 会被 agent 直接绕过。
