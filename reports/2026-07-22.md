### 1. Kastra · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/kastra)

**定位**：给 AI coding agent 加一层 sub-ms 的运行时授权网关。工具调用、
prompt、输入输出四个 hook 位统一按策略拦截，同一份策略跨 Claude Code /
Cursor / Codex / OpenClaw / Anthropic SDK / OpenAI SDK 执行。

**差异点**：市面上做 agent 安全的分两派 —— 一派是 sandbox / 权限白名单
（e2b、firebase functions、docker isolation），偏"隔离容器"；一派是提示
词层加防注入 (Rebuff / LLM Guard)，偏"运行前分类"。Kastra 是第三条路：
在 SDK / agent 客户端和真实 tool 之间插一个策略引擎，按 policy 决定
allow/deny，并且**一次策略多端生效**。这是 Cedar / OPA 的思路搬进
agent stack。

**技术判断**：宣称 <1ms latency + 跨 Claude Code/Cursor/Codex/OpenClaw
四个客户端，只有 SDK-level intercept 才可能，说明他们做了每个客户端的
adapter；tool_use / prompt / input / output 四段都能 hook，实现上应该
是 middleware 而非纯网络代理。工程量不小，engagement ratio 0.377（Top 10
里最高）说明 dev 圈子在认真讨论。

**信号价值**：agent 从 "什么都能干" 走向 "受控执行" 是必然。前一波是
sandbox 化（box / e2b / modal），这一波是 policy 化（Kastra /
MonoCloud）。做 agent 平台的第一年 focus 在"能不能跑"，第二年 focus
在"能不能限"。同一天 Kastra 和 MonoCloud（下面 Top 榜里 191 票的另一
个）同时出货，趋势很明显。

**关注度**：⭐⭐⭐⭐⭐ — Claude Code / MCP 生态最近的 supply-chain
和 tool overreach 问题需要这一层，值得跟一次真实的 policy 语法看看
表达力。

### 2. box · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/box-4)

**定位**：一条 `box new` 命令 2 秒起一台带桌面、SSH、admin 权限的
Ubuntu VM。定价 $0.036/hr，明确对标 E2B / Daytona / Modal 声称便宜 10 倍，
上限 1000 并发。

**差异点**：E2B 走 microVM / firecracker 但按秒计费很贵；Modal 是
serverless GPU 优先，VM 是副产品；Daytona 偏 dev environment。box 直接
只做一件事：agent 需要一次性 disposable 桌面 + shell + admin，那就
用最便宜的方式给。full VM 而不是 container，说明他们更看重 agent 场景里
需要装任意软件 / 改 kernel 参数 / 跑 GUI 应用的需求。

**技术判断**：$0.036/hr 打到 E2B ~$0.5/hr 十分之一，只能靠 KVM 池 +
overcommit 或者 ARM64 baremetal（Ampere / Graviton3）+ heavy oversubscription。
声称 1000 并发 self-served，说明他们做了资源池和 auto-recycle。是不是有
后遗症（noisy neighbor / 存储不持久）要跑 benchmark 才知道，但价格差
一个数量级说明底层架构确实换了。

**信号价值**：agent 侧的经济模型正在从 "GPU compute" 转向 "cheap
disposable CPU"。因为 agent 大部分动作不是模型推理（那部分交给上游 API），
是浏览器、脚本、文件读写、命令行 —— 一堆便宜 IO。谁能把"cheap IO
sandbox"打到 $0.03/hr 谁就赢下 agent infra 底层。

**关注度**：⭐⭐⭐⭐ — 值得试一次真实的 e2b → box 迁移看性能落差。

### 3. AgentManager · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/agentmanager)

**定位**：Claude Code 的"人在环"托盘 app。macOS 上一个 floating window
显示所有 Claude Code session，某个 session 等你回复时冒出来，一键跳到
对应 terminal；全部空闲就自动隐藏。原生 Swift，notarized。

**差异点**：Claude Code 现在流行的做法是开一堆 terminal 平行跑，问题是
它们真的需要输入时你不知道 —— 得来回切窗口。之前的解法是 tmux 里加
bell / notification hooks，或者用 zellij / warp 装 subshell 通知。
AgentManager 是把这层用一个原生 UI 承接：直接在 macOS 层做全局
awareness。**这是一个"围绕 Claude Code 二次开发的 UX 层"，等于承认
Claude Code 本体的多 session 编排还不够。**

**技术判断**：Swift + notarized，7 天试用付费，说明是独立开发者商业化
产品；技术门槛不算高（进程枚举 + tty 状态检测 + NSPanel），但产品品味
决定成败 —— "全空闲时自动隐藏" 这种细节是有做过用户观察的。

**信号价值**：Claude Code / Codex / Cursor 的"多 agent 并行"用法催生了
一整类 cockpit 产品（AgentManager, Remote OpenClaw 也是同一天）。谁做
Claude Code 二次开发的**注意力管理层**（notification、状态聚合、进度
可视化），谁就有机会做成一个小生态位。Anthropic 官方最终会做，但
第三方现在有窗口期。

**关注度**：⭐⭐⭐⭐ — 一直被 "8 个 Claude session 一起跑但漏了 3 个
input 请求" 困扰的人可以直接试一试。engagement ratio 0.44 说明打中了
痛点。

### 4. Humalike x Hermes · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/humalike-2)

**定位**：一个装在 "Hermes Agent" 上的社交智能插件。一条命令挂进去，
agent 在群聊里就知道"何时发言 / 用什么语气 / 谁是谁"，覆盖 Slack /
Telegram / WhatsApp 群聊。

**差异点**：目前的 chatbot 在 1:1 场景是够用的，但一进群聊就出问题
（要么闷不吭声要么每条都插嘴），本质是没有"社交策略"这一层。
Humalike 把这个策略层单独抽出来做成插件，而不是又造一个新 agent，说明
**Hermes Agent 已经在做插件生态**，Humalike 是里面第一个第三方插件
之一。这个产品的信号价值大过它本身：agent framework 从 "写完就完" 走向
"平台 + 插件"。

**技术判断**：能"记住谁说什么 + 判断何时插嘴"至少需要 (1) 每个用户
一份 memory embedding + (2) 一个 "turn-taking" 分类器决定是否 speak。
两者都不是新东西，但整合成一个"drop-in"插件，且能跨 3 个 IM 平台，
说明 Hermes 的插件 API 抽象足够高层。

**信号价值**：agent 生态开始 fork "Chrome Extension" 模型 —— 主 agent
+ 第三方能力包。谁定义了插件协议谁就是平台。Hermes 押的位置和
OpenClaw、Claude Skills 是同一层，值得同时跟三个协议看谁会赢。

**关注度**：⭐⭐⭐⭐ — 榜首 502 票 + 95 条评论说明社区在讨论
"Hermes 生态"这个新叙事，值得单独查一次 Hermes Agent 是谁做的以及
它的插件规范长什么样。

### 5. Arkor · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/arkor)

**定位**：TypeScript 版的 open-weight model fine-tune + 部署平台。让
Claude Code / Codex 直接生成数据集和训练脚手架，本地 studio 点 "Run
Training"，Arkor 云端跑训练并部署成 OpenAI 兼容 API。

**差异点**：目前 fine-tune 主流路径是 (1) Hugging Face + Python 全套自建，
(2) Together / Fireworks 的托管 fine-tune，(3) OpenAI / Anthropic 的
managed fine-tune（能力有限）。Arkor 定位到 "TypeScript 全栈开发者"，
把 fine-tune 变成一个像用 Vercel 一样的动作 —— 明确不是 ML 工程师的
工具。这个用户画像很聪明：正在用 Claude Code 写 app 的人，未来会想给
自己的 app 蒸馏一个小模型，但不想学 Python 全套。

**技术判断**："让 Claude Code 生成数据集 + 训练项目"，说明他们做了
CLI + Claude Code MCP tool + 本地 studio 三件套；训练本身跑在云上但
调度是本地控制。open-weight only（大概率是 Llama / Qwen 系）+ OpenAI
兼容 API，工程闭环合理。真正的问题是训练成本能压多低、以及是否能
支持 LoRA / QLoRA 等低成本方案。

**信号价值**：这是 "fine-tune 民主化" 的一步：把工具从 ML 工程师手里
搬到 app 开发者手里。前一波是 RAG 民主化（LangChain / LlamaIndex），
这一波是 fine-tune。搭配 open-weight 模型质量在上升，未来两年 "私有
小模型 embed 到 app 里" 会是常态。

**关注度**：⭐⭐⭐⭐ — TypeScript 生态里 fine-tune 一直是空白，
值得 star 观察定价和真实训练速度。
