### 1. Cursor for iOS · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/cursor) | [官网](https://www.producthunt.com/r/KBJ3E75U6E6GTF)

**定位**：Cursor 的原生 iOS 客户端（公开 beta）。核心不是「在手机上打字改代码」，
而是**远程操控 coding agent**：手机端可以起云端 always-on agent，也可以远程接管
桌面上正在跑的 Cursor session；agent 干完活推通知，你在手机上 review、合 PR。

**差异点**：GitHub Mobile / Copilot Chat mobile / Warp mobile 这一批都还是「让你
在手机上处理已有 workflow」；Cursor 反过来把手机当成 agent 的**遥控器 + 通知
栏**——桌面继续是 heavy computation 层，手机负责 kick-off 和 approval。类似的
思路 Codespaces + Copilot 曾经想做但没做透，Cursor 直接把「agent 是长期在跑的
后台进程」这一前提写进产品形态。

**技术判断**：真正的工程含量在同步层——cloud agent 的 session 状态、桌面 Cursor
的远程 attach 通道、桌面/手机双端的 diff review UI。不是套壳。iOS native 也意味
着他们在长期押注「agent 通知 → 移动端 approve」这种交互，会碰到后台通知/推送/
授权刷新一整套 mobile-native 的老问题。

**信号价值**：**agent 的默认交互从 IDE 移到了「通知栏 + 移动端 approve」**。
这条路径如果走通，coding agent 的 UI 中心会从编辑器窗口漂到「你正在做的任何事
中间弹出的那个 prompt」。做 AI agent 或 devtool 的方向应关注两点：一是
approval flow 的延迟/信任模型，二是"cloud agent"作为服务形态（不是本地进程）
如何计费。Copilot Workspace / Devin / Claude Code cloud runner 会正面撞上。

### 2. Brain2Qwerty v2 · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/meta) | [官网](https://www.producthunt.com/r/OZ4VDM6D44VI3Z)

**定位**：Meta 出的**非侵入式脑机接口**，从 MEG（脑磁图）信号直接解码句子，
end-to-end 深度学习 + LLM 加持，词准确率 **78%**，不用手术。

**差异点**：BCI 圈子的现有玩家分两派——Neuralink / Synchron / Precision 走
侵入式电极，准确率高但门槛是开颅；Emotiv / Muse / OpenBCI 走消费级 EEG，装置
便宜但信号噪声大只能做「疲劳/注意力监测」这种粗粒度信号。Meta 把 MEG（信号比
EEG 干净得多，但传统设备是房间大小的液氦装置）+ 深度学习组合起来做 sentence-
level 解码，78% word accuracy 在非侵入式里是很硬的数字。上一代 Brain2Qwerty
只做单字符，v2 已经是句子级，一年内跨了一个阶段。

**技术判断**：核心工程在信号处理 + language model prior 的耦合。MEG 硬件不是
消费级——公开的 demo 更像研究平台。Meta AI research 在推的是「BCI 是 LLM 的
新输入模态」这个叙事：语言模型的先验强到可以把稀疏、噪声大的神经信号补全成
自然句子。这是 LLM 反向赋能 BCI，而不是 BCI 赋能 LLM。

**信号价值**：**LLM 正在把非侵入式 BCI 从「玩具」拉到「原型可用」**。做
novel I/O modality 的应该盯住这条线：一旦硬件小型化（MEG 到 OPM 阵列的技术
路径已经在走），"想什么写什么" 会先在 accessibility 场景里落地，之后才是通用
输入。对 AI agent 方向的启发是——**你的输入延迟约束可能从 100ms 打字变成
1s 神经解码**，UI 需要重新设计。

### 3. Dayflow · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/dayflow-4) | [官网](https://www.producthunt.com/r/HN53MKBPHQIQHU)

**定位**：**local-first macOS 应用**，用屏幕内容 + AI 自动写工作日志，专门为
「年底晋升要举证做了啥」这个场景服务。MIT license 开源，模型可选本地或云端。

**差异点**：Rewind、Reflect、Screen Studio 这一批「screen memory」产品此前
都往 productivity / knowledge 方向卷——Dayflow 把叙事聚焦到**晋升举证**这个
极窄的 use case，非常清楚它面向的是大厂高级工程师、经理这种「文档化 impact 决定
下一个 title」的人群。开源 + 本地推理是必要条件（屏幕数据太敏感，云端不可能
接），但选题上的清晰是它的区分度。

**技术判断**：local-first + 支持本地模型 + MIT 是三个都不便宜的选择。屏幕截取
+ OCR + 分段聚合 + LLM 总结的流水线是标准 pattern，但把这套东西做成开源、模型
可插拔、真的能跑本地，工程量在存储管理和推理调度上。产品最难的部分是**摘要的
时间粒度和抽象层次**——把「点了 20 个链接」压缩成「debug 了 auth token 刷新
bug」需要 planning-level 的抽象，这块看运行效果。

**信号价值**：两条：一是 **local-first + 本地推理已经是「screen memory」这
一品类的默认要求**（隐私门槛过硬），二是 **AI 工具正在从「让你效率高」显性
转向「让你能证明你效率高」**——绩效叙事化本身正在被产品化。做 devtool 方向
可以关注「工作行为 → 结构化叙事」这条链条的可扩展性，本质上是给自己写
retrospective 的 agent。

**差异点**：Warp 之类的终端里已经有一些 agent status，Raycast 有 quick access，

**技术判断**：技术上不难——本质是聚合 Claude Code / Codex 的日志或 IPC，做

的 hook 和 CLI 输出，Codex CLI 也有，但如果作者只 hardcode 这两家，一旦
Cursor / Windsurf 这一堆也起 agent，接入面会爆炸。看它有没有做通用的 CLI/log
tail 抽象。

生活方式」的边角工具（switcher、diff aggregator、跨 agent 记忆同步等）。

### 5. Foresight by Lightning Rod · ⭐⭐⭐
[PH](https://www.producthunt.com/products/training-data-generator) | [官网](https://www.producthunt.com/r/ERAFMCUYCEPACK)

**定位**：**OpenAI 兼容的预测 API**——问一个关于未来事件的问题，返回一个
校准过的概率预测，宣称在真实结果上做了 benchmark 评估、比通用 LLM 便宜。
面向 agent、预测市场机器人、决策工具的开发者。

**差异点**：直接 prompt GPT / Claude 问「X 会发生的概率」有个已知问题——LLM
的概率输出没有校准（confidence 和实际频率不匹配），而且对 base rate 敏感。
市面上校准过的预测服务此前基本只有学术产品（Metaculus API、Manifold 数据）或
prediction market 后端。Foresight 直接封装成 OpenAI 兼容 API 卖，是把「校准
预测」当成 LLM 之外的一个原语来兜售——姿势是对的。

**技术判断**：关键在于「trained and evaluated on real-world outcomes」到底
意味着什么。真做校准需要一个持续 resolve 的事件数据集 + Brier score 之类的
持续评估。有可能的路径：(a) fine-tune 一个 LLM 在历史预测赛数据上做 SFT + RL；
(b) LLM + web search + 一个显式的贝叶斯或集成层做 aggregation。宣传里没细节，
风险在于 benchmark 会不会是自己挑的数据集。

**信号价值**：**「校准」正在被抽象成 agent 工具箱里的一个独立能力**——和
web search、代码执行、RAG 平级。做 AI agent 的可以留意：如果 forecasting API
真的比通用 LLM 便宜且校准更好，agent 在做涉及"未来"的决策时会开始 route
到这类专用 API，就像 code interpreter 之于计算。会不会成规模看 pricing 和
benchmark 透明度。
