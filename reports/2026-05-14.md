### 1. Notion Developer Platform · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/notion) | [Website](https://www.producthunt.com/r/POMCHNCCY4J4RS)

**定位**：Notion 不再只是一个内容容器，而是把自己改造成 agent 的
runtime——CLI、Workers、database sync、agent tools、webhook trigger、
MCP server、External Agents API 一次性铺开，让外部 agent 可以把
Notion workspace 当成共享状态层来读写。

**差异点**：和 Notion AI（Notion 自家 LLM 封装）这条线明显切分——
Notion AI 是"在 Notion 里用 AI"，Developer Platform 是"让别人的 agent
来用 Notion"。最近的对位是 Linear 的 Agents（Linear 把 issue 系统
开放给外部 agent 编辑）和 Slack 的 AI Agents API，但 Notion 押的更
重：它一次给了 7 个原语（CLI / Workers / DB sync / agent tools /
webhook / MCP / External Agents API），把 Notion 直接定位成"AI
workspace 的标准接口"。这是 SaaS 巨头第一次把自家文档系统 **以 MCP
和 External Agents 协议为一等公民** 重新封装。

**技术判断**：External Agents API 是关键——这个名字暗示了和
Anthropic / OpenAI 的 agent SDK 走相同协议（callbacks / structured
tool calls），而不是 Notion 自家 webhook 协议。Workers 看起来是
"在 Notion 侧跑用户代码、订阅 workspace 事件"的 serverless 形态，
对标 Slack Workflow Builder + Cloudflare Workers。MCP server 应该
就是把 page / database / block 操作暴露成 MCP tools，让 Claude
Code / Cursor / 任何 MCP client 都能把 Notion 当 backend。这一套
落地后，Notion 的护城河就从"UI" 转到"workspace schema"——schema
被 agent 调用得越多，迁移成本越高。

**信号价值**：这是 productivity SaaS 巨头里第一个**把 MCP 当作官方
平台一等接口**放出来的（之前都是社区第三方 MCP server）。这意味着
2026 下半年其他 workspace 产品（Linear、Asana、ClickUp、Coda）大概率
跟进，"打开 MCP 接口"会变成标配。对做 agent 框架的，意味着可以把
"connect to user's Notion / Linear" 从 OAuth + 自写 REST wrapper
降级到"一行 MCP config"——agent 集成成本会陡降，竞争点从"接得通什么"
转到"接通之后做什么"。

### 2. Raindrop Workshop · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/raindrop) | [GitHub](https://github.com/raindrop-ai/workshop)

**定位**：本地开源的 agent debugger——你跑 agent 时它在本地 stream
出 token 级 trace，另一个 agent（Claude Code）通过 MCP 读这些 trace，
然后写 eval、replay、修 bug、再跑一遍。作者把这个循环叫
"Self-Healing Agent loop"。

**差异点**：现有 agent observability 这条线（LangSmith / Helicone /
Langfuse / Arize）都是**云端 dashboard 给人看**，给人提供"看完之后
手动改 prompt"的工作流。Raindrop 反过来做了两件事：（1）trace 数据
留在本地，不上传；（2）trace 的消费者是另一个 agent，不是人。
这就把"调 agent" 从"人看 dashboard → 改 prompt" 改成"agent 看
trace → agent 写 eval → agent 改 agent"。开源 + 本地这两点是
LangSmith / Helicone 没法跟的——它们的商业模式建立在 trace 必须
上云这一假设上。

**技术判断**：能 stream token 级 trace 且让外部 agent 通过 MCP 读，
意味着架构上：（1）本地 agent runtime hook（很可能是 LangGraph /
LangChain / 任意 stream 接口的 wrapper），把 token / tool call /
state 写到本地 ringbuffer，（2）一个 MCP server 把 ringbuffer
投影成 query tool（`get_trace(run_id)` / `list_failures()` /
`replay(span_id)`），（3）一份 eval template 让 Claude Code 知道
怎么针对失败 trace 写 assertion。难点应该在 trace 数据结构稳定
（不同 framework 的 span 语义对不齐）和 replay 决定论
（外部 API 调用怎么 mock）。

**信号价值**：把 **MCP 作为"agent 互相协作"的标准接口**，这是今天
最纯粹的一个例子——不是给 agent 接工具，而是让 agent A 去读 agent B
的内部状态。这是 agent-of-agents 架构的一个关键拼图：observability
本身变成可被 agent 调用的 service。对做 agent 框架的，值得把 trace
schema 升级成 MCP-native（不只是给 dashboard 用），下一阶段大家
比的是"我的 agent 能不能被另一个 agent 高效调试"。

### 3. Asteroid · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/asteroid-3) | [Website](https://www.producthunt.com/r/IZAH4P7DZJMI5S)

**定位**：computer-use agent 的"meta-builder"。你给它描述任务（自然
语言），它自己组装一个跑在 browser / Linux / Windows 上的 agent，
**边跑边把动作固化成 script**，下一次直接跑 script 而不是重新让
LLM 决策。已经在 EHR / 保险 / VPN 内网这类难场景下完成 15 万+次
执行。

**差异点**：computer-use agent 这条线（Anthropic Computer Use /
Claude for Desktop / Browserbase + Stagehand / Multi-On）大多在两个
极端：要么是"裸 LLM 看屏幕点东西"（贵、慢、不稳定），要么是
"RPA 录脚本"（稳定但写起来痛苦）。Asteroid 走中间路线——**先用
LLM 探索，跑通了之后固化成 deterministic script**。这其实是 RPA
2.0 的正确形态：用 LLM 当 onboarding cost，让生产路径回到便宜的
脚本执行。最接近的对位是 Reworkd / Skyvern，但 Asteroid 把
"meta-agent 写 agent"这个抽象层（Astro）做成独立产品形态而不是
implementation detail。

**技术判断**：15 万次/月 across EHR / Citrix / VPN 这种被 RPA 厂商
（UiPath / Automation Anywhere）锁了 20 年的场景，意味着他们在
selector 稳定性、auth / 2FA、长 session 维持上做了大量工程——
这部分是 demo 看不出来的。"meta-agent 把动作变 script"暗示是
**hybrid execution**：一条任务的不同 step 可能跑在不同模式
（脚本 vs LLM），fallback 路径自动重新进入 LLM 探索。如果做得好，
单 run 成本应该比纯 computer-use 低一个数量级。

**信号价值**：computer-use 这块**正在从"AI 演示品"过渡到"企业自动化
新标准"**——15 万次执行的体量说明已经有 paying customer 在 EHR /
保险这类强合规场景下用。配合今天的 Open Browser Use / Open Computer
Use，"computer use 基础件开源 + 应用层企业化"这条剪刀差已经出来了。
对做 RPA 替代 / 内部自动化的，是时候停止"我要不要等 computer use
稳定" 的犹豫——它已经在生产了。

### 4. Open Browser Use · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/open-browser-use) | [GitHub](https://www.producthunt.com/r/YE6HTLXLTJLIXI)

**定位**：把本地 agent 接进**你真实的 Chrome profile**——MV3 扩展 +
native host + CLI + MCP server + JS/Python/Go SDK。让 agent 能开
tab、claim tab、跑 CDP 命令、看 download、处理 file chooser，全程
不走任何托管的 browser 自动化服务。

**差异点**：浏览器自动化这条线长期被两类方案夹住：（1）Playwright /
Puppeteer 这类**新建 headless 浏览器**——和你日常用的 profile 隔离，
登录态、cookie、扩展都拿不到；（2）Browserbase / Stagehand / Anchor
这类**云端托管浏览器**——profile 也在云上，敏感数据要先上传。Open
Browser Use 走第三条路：**MV3 扩展驻留在你日常 Chrome 里，agent
通过本地 native messaging + MCP 接它**——profile 留在本地、登录态
留在本地、cookie 留在本地。这是浏览器 agent 最自然的部署形态，
但因为 MV3 + native messaging 的工程门槛一直没有 OSS 实现。

**技术判断**：MV3 + native host 这个架构选择有几个隐含承诺：
（1）能绕过云端浏览器的"必须把 cookie 上传"问题（重要的安全
卖点）；（2）能复用用户已经登录的 SSO（这一点对企业内网
agent 是 game-changer）；（3）CDP 命令直接走 Chrome 原生通道，
不通过任何远程 driver。MCP server 这一层意味着 Claude Code /
Cursor / 任何 MCP client 都能 zero-config 控制本地浏览器——
对开发者 agent 的"打开 PR 页面看看 review 评论"这类轻量自动化
是巨大的便利。

**信号价值**：和 Asteroid / Open Computer Use 一起，今天可以确认
**"local-first computer-use" 已经从概念变成具体产品形态**。
和 Raindrop 一样押注"重要的 agent infra 必须本地 + 开源"。
做 browser agent 应用层的，可以把 "需要用户的真实登录态" 从
障碍变成卖点——OBU 这层把它降到 install 一个扩展。

### 5. Edit Mind · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/edit-mind-3) | [Website](https://www.producthunt.com/r/ZDMML5RKVQG7LY)

**定位**：100% 本地的视频内容检索——给视频编辑、记者、内容创作者，
**像跟朋友描述场景一样**搜几小时素材。不切换剪辑软件，不上云。

**差异点**：video search 这条线（Twelve Labs / Reduct / Iconik /
Frame.io 的 AI search）都需要把素材上传给云端的 multimodal 模型。
对剪辑 / 新闻 / 后期这类工作流是个硬伤——版权未发布素材、客户
NDA、敏感访谈都不能上传。Edit Mind 走纯本地路径，对位最像
**Recall + Whisper local + CLIP local**，但做成了剪辑师工作流原生
的产品形态（不离开 Premiere / FCP）。在视频检索这个具体垂直里，
"本地"是个非常硬的产品差异，不是"nice to have"。

**技术判断**：要在 Mac 上做本地 multimodal 视频检索，技术栈基本
锁定：CoreML / MLX 跑 vision encoder（CLIP / SigLIP / 自家蒸馏
小模型）+ Whisper local 转录 + 本地 vector index（DuckDB-VSS /
LanceDB / 自研）。难点不是模型，是**索引规模**——几小时素材
×多分辨率 frame embedding，可能轻松上 GB；以及**索引增量**——
导入新素材时怎么不重做全量。从 PH topics 没有列 "AI"
（只有 mac / github / video）也是个有意思的信号：他们把自己定位
成"Mac 上的视频工具"，AI 是实现手段不是卖点。这是 on-device
inference 已经成熟到不需要标榜的标志之一。

**信号价值**：on-device multimodal 推理跨过临界点——**不再需要在
营销里强调 AI**，能直接说"100% local"作为差异化。这一年里
Whisper / CLIP / SigLIP 的小模型版本性能已经够民用应用使用，
配合 Apple Silicon 的内存带宽，类似 Edit Mind 这样的"垂直工作流
本地工具"会成片出现。下一波可能是音频（podcast editor 的本地
检索）、设计稿（Figma file 的本地语义搜索）、PDF 库
（local Mendeley + LLM）。对做 dev tool 的也是个提示——不是
所有问题都需要 cloud LLM。
