### 1. Emdash · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/emdash) | [Site](https://www.producthunt.com/r/U7NUCTTEX6BM3I)

**定位**：开源桌面 app，把 Claude Code / Codex / Cursor / Aider 这些
coding agent 放在同一个窗口里并行跑——监控会话、对比 diff、把 GitHub
issue 一键转 PR。

**差异点**：和昨日 (5-19) 的 ShioriCode 撞品类，但 Emdash 明确押注
**并行多会话**和 **issue → PR 工作流**，而不是只做 GUI 包装。换句话说
它定位的不是「Codex 的好看版」，而是「我同时让 3 个 agent 在不同分支上
跑，最后挑一个最好的」——这是 Anthropic/Cursor 最近在内部演示里反复
强调的"swarm" 工作流，第三方第一次把它产品化。其他参照系：sketch.dev、
Devin Multi、Conductor。

**技术判断**：从描述看是 Electron/Tauri + 各 CLI agent 的 wrapper，但
要把 diff review、session 状态机、PR 提交链路做稳，工程深度其实不浅。
是不是套壳取决于 conflict resolution、并发 file lock、agent-level
permission scope 这些细节有没有解。开源是关键加分项——可以直接读代码
验真伪。

**信号价值**：**agent orchestration 正在从 "frame" 走向 "fleet"**——
一年前是「我配一个最好的 agent」，今天是「我同时让 N 个 agent 比赛」。
对做 agent 工具的人来说，下一层产品形态可能不是更聪明的单 agent，而是
比较框架 + 协作机制 + 仲裁器。Emdash 这种产品如果能跑通，会反过来给
底层模型公司加压：API 要支持 better session multiplexing、cheaper
parallel sampling、stateful checkpoint restore。

### 2. Re_gent · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/re_gent) | [Site](https://www.producthunt.com/r/YYIC7WP7TME3QF)

**定位**："Git for your AI agent's actions"——把 agent 在你的代码库
里做的每一步动作拎出来，按 prompt 维度版本化、可追溯、可回滚（跨
文件、跨会话）。

**差异点**：现有 agent 工具的 undo 都是单文件粒度（IDE 的 Ctrl+Z），
或者会话级别（重开一个 Claude/Cursor session）。Re_gent 把"**agent
做的事**"当成 first-class object 来做 VCS——按 prompt 切片、聚合
跨文件的副作用、可以选择"撤销这次 prompt 引发的所有改动，但保留之后
我手写的部分"。这种粒度的产品我**之前没见过**——既不是 git，也不是
agent memory，是一层全新的 audit / control plane。

**技术判断**：核心难点是 (1) hooking 到各个 agent 的执行流（Claude
Code、Cursor 都有 hooks SDK，可以拦截 tool calls），(2) 设计一个比
git tree 更适合表达「prompt → multi-file diff → user manual edit」
的数据结构。开源 + GitHub topic 说明他们在做 plumbing 而不是套壳。
真正的护城河会是 hook 覆盖面 + 撤销算法的正确性。

**信号价值**：这是今天最值得长期跟踪的一个。**agent 越强，可观测 +
可干预的工具就越值钱**——这条规律在过去 12 个月已经在
LangSmith / Helicone / Braintrust（observability）和
Continue / Cline （inline control）两条线上都验证过。Re_gent 是
**第三条线：agent action 的 source-of-truth 层**。如果这个抽象站得住，
未来所有 coding agent 都需要接入它，类似 dotenv 之于 secrets、
git 之于 commit。对做 agent 产品的人来说，这是值得**先用一周，再
决定要不要在自己产品里 native 集成的等级**。

### 3. Glia · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/glia-2) | [Site](https://www.producthunt.com/r/ZJAI6VMV2IIXAG)

**定位**：100% 本地 + 开源的 agent 记忆桥——Chrome 插件抓你和
Claude/ChatGPT 的网页对话，**写到本地共享 SQLite**；同机的 MCP server
让 Cursor / Claude Code 可以**查这些决策**。

**差异点**：市面上 agent memory 工具几乎全是 SaaS（mem0、Letta、
Zep），数据在云上、collection 通过 API。Glia 选了完全相反的路：**本地
SQLite + MCP**，没有云端、不需要登录、不上传任何对话。同类里只有
Contextberg（今日同期发布，96v）思路最近，但 Contextberg 是被动屏幕
监控、范围更宽，Glia 是定向抓 LLM 对话、更精准。

**技术判断**：技术上不深——Chrome extension 抓 DOM + 本地 sqlite + MCP
server。但**架构选择**非常正确：MCP 是这半年新出现的协议（Anthropic
2025 中推出），用 MCP 当本地内存接口是最低摩擦的接法，IDE 端不用任何
适配。开源 + 100% local 是真正的差异点，不是噱头。

**信号价值**：今日**第三个**绑定 MCP 的产品（Contextberg、Skilled
也都是）。MCP 在 5 月份明显进入「**第三方默认集成**」阶段——之前是
Anthropic 自家在推，现在开发者把它当成 "USB-C for AI tools" 那种
基础设施在用。对做 agent / 开发者工具的方向：**MCP 端口设计已经不是
"要不要支持"，而是 "MCP server 是你的产品的一等公民还是附属物"**。
本地优先这条线（Glia / Invenio / Skilled）也在今天显著抬头，是
privacy + cost + latency 三重驱动的副产物。

### 4. Runtime · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/runtime) | [Site](https://www.producthunt.com/r/TOZOGOJYMEHTNQ)

**定位**：把 coding agent 包装成"队友"——Slack / Linear / CLI / 浏览器
都能调，运行在沙箱里，自动注入公司 context、权限、集成。

**差异点**：和 Emdash 都是 agent orchestration，但定位完全不同。
Emdash 是给**个人开发者**的并行执行器，Runtime 是给**团队**的共享
入口（PM 在 Slack 里 @ runtime "查一下上周 retention"，自动有 SQL +
仓库权限）。和 Devin、Cosine 同一象限，但他们都是 closed agent；
Runtime 强调"**bring your own agent**"——只做沙箱 + 集成 + 权限层。
和 GitHub Workspaces / Coder.com 那种远程开发环境也不同——Runtime
是 agent-first 的，UI 是聊天/issue，不是 VS Code。

**技术判断**：核心是企业级沙箱（要做到资源隔离、网络出站策略、密钥
注入）+ 权限映射（agent 拿到的 token 跟发起人挂钩）+ 多入口集成。
工程量很大，但创新点不在技术——在产品定位：把 agent 从 IDE 里挖出来
放进 Slack，让非工程师也能用。这件事 Anthropic 的 Claude in Slack、
Glean 都试过，Runtime 的差异是**专为代码协作**。

**信号价值**：**Agent 的 "team product layer" 正在成形**。过去半年
agent 都是 IDE 内的工具，对应单兵作战；下一步是变成共享资源（公司
SQL 助手、PM 自助构建仪表盘、Support 自动改 bug fix）。这要求权限、
计费、可观测、安全审查打包成一层 IT-level 的基础设施——Runtime 在押
这一层。竞争路径会很拥挤（Cosine、Devin、Sweep、Anthropic 自家
Claude for Work 早晚下场），但**整个品类还在很早期**。

### 5. Viberia · ⭐⭐⭐
[PH](https://www.producthunt.com/products/viberia) | [Site](https://www.producthunt.com/r/EQOGZXTYHXDDA2)

**定位**：把你的 AI agent 团队画成**等距俯视图**（像玩文明 6），每个
agent 是地图上的一个角色，状态图标标示「在忙 / 卡住 / 等你」，点进去
可以聊、换模型、挂技能。

**差异点**：今日所有"agent 管理器"都是**列表 UI**（Emdash / Runtime /
Re_gent / Glia 都是窗口 + 标签页），Viberia 是唯一押"**空间隐喻**"的
——文明 6 / Civilization 风格的 isometric map。这不是噱头：当你管理
20 个 agent 时，列表 UI 会失效（眼睛追不过来），人类大脑对**空间位置**
的并行处理远胜于线性扫描。这是一个真实的 UX 假设。

**技术判断**：底层应该是 Electron + Three.js / Pixi（isometric tile
rendering）+ 多 LLM 后端切换。空间布局 + 状态机的工程不浅，但还达不
到深度技术护城河。最大的风险是「好看但不实用」——如果 90% 时间用户
就是想 chat，那地图就是负担。

**信号价值**：**新型 I/O 是 CLAUDE.md 的明确关注点**，Viberia 是难得
的「**不是 chatbox**」的 agent UI 尝试。Codex Cloud 那种 dashboard
是工程师味的列表，Viberia 是消费级游戏味的画布——这两种走极端的形态
比中间地带（"会话列表 + 状态栏"）更可能跑出意外的产品形态。值得短期
试用，看是不是「好看但效率低」一票否决。如果他们能给空间布局加上
"自动按相关度聚类"这样的语义层，会从噱头变成真有价值的 UX。
