### 1. Shadow 2.0 · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/shadow-6)

**定位**：实时会议执行 agent。它不再只做"会后总结 + 待办列表"——而是在
会议进行中识别意图、立刻调外部工具完成（生成 PDF、做 slides、写 CRM、
发跟进、订日程），目标是会议一结束 to-do list 已经清零。

**差异点**：和 Granola / Otter / Fireflies 这一代"会议笔记 + 摘要"工具
正面区分——它们停在"摘要 → 人类执行"，Shadow 把这条链压缩成"实时识别
→ 自动执行"。和 Spinach / Krisp 的 action-item 提取也不是一回事，那些
仍是建议层，Shadow 是真的去 connector 那边把动作做了。

**技术判断**：栈大概是 (1) 实时 ASR + 滚动语义解析 (2) 一组
CRM/邮箱/日历/文档 connector (3) "什么时候触发什么动作"的判定层（最难
的一层，错触发比漏触发代价高）。从 2.0 命名看 1.x 已经踩过执行可靠性
的坑。163 评 / 0.34 ratio 是今日讨论度第二高，说明用户真正在追问执行
边界、撤销机制这些点，是有工程信号的发布。

**信号价值**：会议工具赛道集体从"摘要"往"执行"迁移。摘要不再是壁垒
（任意 LLM 都能写），能不能闭环执行才是。下一波生产力 agent 大概率
都长这样：识别 → 直接动手，把"人类做最后一步"砍掉。做 agent 编排
方向的可以盯一下他们的 trigger / undo / consent 设计。

层的非工程人员也能直接用。开源是关键差异点：跟 Glean / Sana 这种
封闭 enterprise context layer 把竞争维度从功能换成可控性。

**技术判断**：220 评 / 0.46 ratio 是今日最高，但要区分"产品引发讨论"
和"发布噱头催评论"——要看 GitHub 才知道 schema、permission、
agent 写入回流是否真有设计。tagline "for you, your agent, your
coworker and their agent" 把"双向使用"当作核心命题，至少叙事上是
认真的。

**信号价值**：team-context 正在被识别为 2026 年 agent 编排的关键
缺失层——单 agent 工具够多了，但"团队多个 agent 共享同一上下文"

### 3. Superset 2.0 · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/superset-5)

**定位**：在任意机器（本地 / 远程）上并行跑 100+ CLI coding agent
的编排平台。2.0 是从头重写，加了远程 workspace 和团队实时协作。
"Works for any CLI agent"——不绑 Claude Code 也不绑 Codex。

**差异点**：和 Cursor / Windsurf 这种 IDE 内 agent 不在同一形态——
它不是"我和一个 agent 对话"，而是"我同时监工 100 个 agent"。最近
Conductor / AmpCode 也在这条赛道上，Superset 的差异在于 (a) 显式
覆盖远程机器（不只是本地容器）(b) 不绑特定 CLI 实现，把自己定位成
fleet manager 而非 agent runtime。

**技术判断**：远程 workspace 大概率是容器化 sandbox + PTY/工作流
代理（这样 any CLI agent 都能挂进来）+ 实时协作（CRDT 或 ot 之类）。
重写理由说得通——v1 多半把 UI、编排、sandbox 耦合死了，加远程
就要重做底盘。"任意 CLI agent 兼容"是合理的押注，因为 IDE 内 agent
锁定客户，CLI agent 的客户来去自由，刚需 fleet 工具。

**信号价值**：今天最重要的范式信号——"1 个开发者 = N 个并行 agent"
继续具象化。一旦这个工作模式跑通，瓶颈从"agent 能力"转移到"开发者
注意力"，UI/UX 会从单 agent 对话窗口转向多 agent 监工面板（任务
分发、状态聚合、冲突解决、提交审批）。做开发者工具的方向，"管 agent
的 agent"是非常确定会出现的下一层。

### 4. pay.sh · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/solanafndn)

**定位**：与 Google Cloud 合作的开源实时支付基础设施，让 AI agent
自主发现、调用、按次付费给任意 API。直接把"agent 当付费方"作为
一等公民。

**差异点**：现有 API 计费 / 支付方案（Stripe Metered Billing、AWS
Marketplace、RapidAPI）以"开发者签合同 + 拿 key"为前提。pay.sh 把
这套链路做成 agent 透明：发现 → 鉴权 → 付费 → 调用，全部不需要
人类介入。同期赛道里还有 Skyfire、Payman、PayPal Agentic
Payments，pay.sh 的差异在 (a) 开源 (b) 跟 Google Cloud 走，意味着
分发位置好。

**技术判断**：URL slug 是 `solanafndn`（Solana Foundation），
强烈暗示底层结算用 Solana 链上做（低费 + 秒级最终性匹配 per-call
付费）。但 tagline / description 完全没提区块链——明显在做"对 agent
透明的 settlement 层"，不让用户感知。Google Cloud 大概率负责 discovery
+ routing 侧。难点不在链上，在 (1) agent 信用 / 限额 / 撤销的策略层
(2) 微小金额（$0.0001 / call 量级）的工程化 (3) 与 OAuth / API key
现有模式的兼容路径。

**信号价值**：这是今天战略意义最大的发布。如果 MCP 是"agent ↔ tool"
的连接协议，pay.sh 想做"agent ↔ 有偿 API"的结算协议——同一层级的
基础设施野心，但更难（涉及钱）。agent 经济学开始有专门支付层，意味着
2026 年的关键问题不再是"agent 能不能干活"，而是"agent 怎么自己买
原料"。做 agent infra 的方向必看。

[PH](https://www.producthunt.com/products/wozcode)

**定位**：Claude Code 的"效率寄生层"，两条命令装上去之后，不换 IDE
/ 不换订阅 / 不换工作流，号称砍 50% token 消耗。

**差异点**：和 Aider / Cline / Continue 这些是替代关系不同，WOZCODE
明确把自己定位成 Claude Code 的补丁——不抢用户、寄生在 Anthropic
生态里收"省钱税"。这是个少见的合作型生态位。

**技术判断**：50% 节省的常规手段就那几个：context window 裁剪、

走 Sonnet）。一定有 trade-off——少送上下文意味着 agent 决策质量
下降；如果作者声称无下降，那就是 Claude Code 默认 prompt 里有冗余
被砍。技术杠杆清楚但不复杂，工程量在保持兼容性 + 不破坏 tool use 链
路。两条命令安装暗示是 npm 包 / proxy hook 形态。

**信号价值**：Claude Code 用户基数已经大到值得做"节流外挂"——这本身
反向佐证了 Claude Code 的商业体量。也意味着"agent 运行 token 经济学"
正在成为开发者侧的成本痛点，下一波工具会显式打"省 token"卖点（类似
当年 CDN / Webpack tree shaking 的角色）。对做 agent 工具链的方向，
这是个提醒：在 inference 成本曲线没大幅下降之前，会有一整代寄生型
优化层冒出来。
