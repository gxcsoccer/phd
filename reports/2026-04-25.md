### 1. Clawdi · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/clawdi) | 277 votes / 31 comments

**定位**：把 AI agent 的运行环境（memory、API keys、技能集）从具体 agent 框架里剥出来，放在云端做"持久化的 agent 家"。开源，自托管。

**差异点**：今天大多数 agent 工具都是"再写一个 agent runtime"（LangGraph、AutoGen、Crew 各自一套）。Clawdi 反过来——做框架无关的状态层，让你换 framework 时不丢 memory 和 skill。和 LangSmith / Letta 思路接近，但更偏"agent 的 home directory"，加上 chrome extension 入口。

**技术判断**：GitHub + Chrome extension 双链，应该是 web 后端 + 浏览器宿主。"open-source environment that decouples …" 这个抽象方向是对的，但 31 条评论里需要看是否有人吐槽是壳——明天回看。

**信号价值**：agent 状态管理正在和 agent 引擎解耦，是个明显的趋势（参考 OpenAI Memory、Letta、Mem0）。如果你在做 AI agent 平台，"用户的 agent 状态"会成为下一个粘性点而不是 agent 本身。

### 2. MiMo-V2.5 Voice · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/mimo-v2-5-voice) | 117 votes / 3 comments

**定位**：小米开源的 8B 双语 ASR 模型，专门处理普通话 + 英语 + 八种中文方言 + code-switching + 歌词转写。

**差异点**：Whisper-large-v3 在方言/code-switching 上一直拉胯，国内 Paraformer 系列又不开源。MiMo 一次把"方言+code-switching+唱歌"三个长尾场景打包，且模型体量只有 8B。如果质量真的可用，会是中文语音工程的一大块缺口被补上。

**技术判断**：8B 参数 ASR，明显走 LLM-style 解码路线（不是经典 conformer），与去年开始流行的 Qwen2-Audio / Phi-4-multimodal 一脉相承。开源 + Github 标签可信度高，不是产品包装。

**信号价值**：大厂在"非英语 + 多语言混合"语音上的开源动作是值得跟的方向。短视频/直播场景里，code-switching ASR 是真刚需，目前几乎没人能做好。

### 3. PromptPaste · ⭐⭐⭐
[PH](https://www.producthunt.com/products/promptpaste) | 116 votes / 3 comments

**定位**：原生 Apple 应用（Mac/iPhone/iPad）做本地 prompt 库，⌘⇧P 全局快捷键秒贴，模板支持 `{{variables}}`，无账号、不上传。

**差异点**：市面上的 prompt 管理工具基本都是 web app + 云同步（PromptHub、PromptLayer 之类），把"我私人的 prompts"放到别人云上其实不舒服。PromptPaste 反过来：原生 app + iCloud 同步 + 全局快捷键，更像 Alfred/Raycast 的工作流而不是 SaaS。

**技术判断**：纯 Swift/SwiftUI 应用居多，技术上不复杂；价值在产品形态本身——把 prompt 当成文本片段（snippet），而不是当成 SaaS 数据。

**信号价值**：local-first + 键盘优先的"个人 AI 工具链"在 Mac 端有清晰 niche。如果你做开发者向 AI 工具，原生 app + 全局快捷键 + 不要账号 是一组复用度很高的卖点组合。

### 4. Euphony · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/euphony-2) | 98 votes / 1 comment

**定位**：把 Harmony JSON/JSONL 对话和 Codex CLI session log 渲染成可交互、可过滤的浏览器时间线，给 AI 工程师 debug agent 工作流用。

**差异点**：现在 debug agent trace 的工具几乎都和某个 SDK 绑（LangSmith → LangChain，Phoenix → OpenInference，Braintrust → Braintrust SDK）。Euphony 直接吃 Harmony 格式（OpenAI 开源的 chat 协议）+ Codex CLI 日志，不绑 framework，这是 OpenAI 把 gpt-oss + Harmony + Codex 推开放后留出的工具生态位。

**技术判断**：1 条评论说明只有内行才会点进来；定位极为精准（gpt-oss / Codex 开发者）。技术上是纯前端可视化，门槛不高，胜在选对了协议层。

**信号价值**：Harmony 格式开始有第三方工具支持，意味着 OpenAI 推 gpt-oss 这条线在缓慢成型。做 AI agent infra 的人值得开始把 Harmony 当成"可移植的 trace 格式"看待。

[PH](https://www.producthunt.com/products/reteai) | 5 votes / 1 comment

**差异点**：和 Ollama / LM Studio 的"单机本地"不同，ReteAi 在"分布式本地"这个角度切入——同一个家庭/小团队的多台 Apple 设备拼算力以跑更大模型。Exo Labs 也在做类似事情但走开源路线，ReteAi 走付费 + 易用 app 路线。

**信号价值**：把"本地推理"从单机升到"家庭级算力池"是一个还没人做明白的方向。买断制 + Apple 生态闭环 是个反订阅的姿态，配合本地 LLM 趋势可能找到一群不愿意月付 ChatGPT 的高端用户。
