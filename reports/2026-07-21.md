### 1. Manifest · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/manifest-363) · 156 votes / 81 comments · **ratio 0.52**

**定位**：把任意网页翻译成"AI agent 能点什么 / 填什么 / 提交什么"的
结构化 JSON。每个 action 附带 resolved CSS / role locator，还有
`requires` 字段编码**元素之间的依赖**（例："必须先选 plan 才能点这个
按钮"）。Python SDK、LangChain 集成、MCP server 一起给。

**差异点**：browser-agent 这个方向过去一年里，主流做法是靠 aria snapshot
+ vision（Anthropic Computer Use、OpenAI CUA、Browserbase、Skyvern
etc.）——都是"扔整个 DOM/截图 给模型，让它猜"。Manifest 反过来：
**先离线把页面结构 + 交互依赖编译成一份 manifest，agent 拿到的是"这里
能做什么"而不是"这里长什么样"**。这是从"感知"到"抽象接口"的收敛，思路
更像给网页做 IDL，而不是给它做 OCR。selector 稳定性 + `requires` 依赖
图两件事，正是当前 browser agent 最难的部分——selector 一改脚本就废，
表单前置条件没建模就无限循环。

**技术判断**：核心难点是**如何在无标注情况下抽取 requires 依赖图**——
需要跑一个 headless 会话反复试探 action 之间的开启关系，或用 LLM 结合
DOM diff 做因果推断。做对了这一步就是护城河；做不对就是又一个 aria
wrapper。Python SDK + LangChain + MCP 三件套齐发说明团队定位很清楚：
**不做 agent，卖给做 agent 的人**。ratio 0.52 是今日 Top 10 里最高，
说明社区（尤其做 browser 自动化的人）觉得这个 primitive 值得讨论。

**信号价值**：**browser-agent 这条线开始出现"共享基础设施"层**。过去
每家 agent 团队都要自建 web 交互抽象，Manifest 想把这一层抽出来做成
Stripe-式的 API 化基础设施。对做 agent 项目的团队直接的启发：**"感知"
不是必须直接对着像素做，先把网页编译成 IDL 是可行路径**。同期 Skyvern、
Browserbase 走的是"感知 + 云托管"路线，Manifest 是"编译 + 客户端集成"，
两条路线之间选哪条决定 agent 架构。

### 2. CartAI · ⭐⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/cartai) · 458 votes / 111 comments · **ratio 0.24**

**定位**：**在任何 live 商家页面完成 checkout 的 agent rails**——
不需要商家侧做集成。一个 dev-first API，切成四件：Catalog（跨商家
搜索 + 实时价格）、Checkouts（下单直到确认）、Payments（Visa /
Mastercard 的 "agent 通道"接 PCI）、Monetization（每笔 agent 成交抽
佣金）。**明确说"cooperate with bot detection instead of evading"**。

**差异点**：这个赛道里的老玩家分两类——一是 shopping agent（Fetch、
Karma、Instacart Order Ahead）都靠**商家 API 或人工爬虫**，覆盖有限；
二是浏览器自动化 SDK（Browserbase / Stagehand / Playwright）都
**只提供工具，不解决 checkout 特有的支付 + 反欺诈问题**。CartAI 做的是
**"支付基础设施 + 商家兼容层"合体**——Visa / Mastercard 出的 "agent
rails" 是 2025 年底才逐步公开的通道（agent 有可核验身份，商家愿意
放行），CartAI 是第一批把这条通道直接封成 SaaS 的。**不 evade
bot detection 的姿态**说明它是走"银行-支付网络-商家"联盟这条合规路径，
不是灰产爬虫。

**技术判断**：真正难的三件事：
1. **支付网络的 agent 身份认证**——需要跟 Visa / MC 的 agent rails
   直接对接，普通团队 6-12 个月拿不下来；这是 CartAI 最硬的护城河。
2. **跨商家 checkout 流程建模**——不同商家 checkout 流程差异极大
   （多步 / 单页 / 3DS / captcha），需要做通用抽象 + 商家指纹库。
3. **Monetization 层的 attribution**——每笔成交怎么算给谁，涉及跟
   商家或支付网络的收入分成协议。

**信号价值**：**"agent commerce"从概念到具体基础设施**。Anthropic 和
Google 上半年都发过 agent 支付协议白皮书，CartAI 是第一个把它做成 dev
API 的产品。做 agent 项目的团队要注意：**"agent 能不能替我下单"从
demo 走向可产品化**——如果 CartAI 通道稳定，接一个 API 就等于让你的
agent 拿到跨商家 checkout 能力，不用一家一家谈集成。反面：**很快
Amazon / Shopify / Walmart 会试图各自出封闭方案锁住流量**，这就是
一个"标准之战"的窗口期。

### 3. CreateOS Sandbox · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/createos-sandbox) · 181 votes / 67 comments · **ratio 0.37**

**定位**：**agent builder 用的 hardware-isolated sandbox，p90 冷启 ~30ms**。
CLI / SDK / Claude plugin / computeSDK 集成 + 50+ 示例一起放出。

**差异点**：直接对标 **E2B、Modal、Runloop、Cloudflare Workers AI
sandbox**。E2B 的冷启在 500ms–2s 量级，Modal 是长驻 warm pool。
CreateOS 宣传的 **~30ms p90 冷启 + 硬件隔离**，如果属实，比现有方案快
一个数量级——这只可能通过 microVM + snapshot restore（类似 Firecracker
的 snapshotter）或轻量 unikernel 实现，不是普通容器路线。硬件隔离而非
namespace 隔离，也说明是给"跑不可信 agent 代码"这个场景做的。

**技术判断**：30ms p90 冷启在 microVM 领域是 aggressive 的数字——
Firecracker 官方 benchmark 也要 125ms 起步；CreateOS 要么是 snapshot
restore 做到位（预热 base image → 分叉），要么用了更轻量的
virtualization（Kata / Nabla / Unikraft）。50+ SDK 示例 + Claude
plugin 说明 GTM 已经想清楚：**主打 agent framework 集成**（LangChain /
CrewAI / Claude Code plugin 生态），不走"通用容器"这条烂大街路线。

**信号价值**：**"给 agent 跑代码"这个基础设施开始出现速度维度的竞争**。
过去 12 个月 E2B 是事实标准，但它的冷启会成为高频短代码 agent（agent
每 turn 都要跑一段 python）的瓶颈。CreateOS 的赌注是**"agent 生态足够
大，speed matters"**。要不要用取决于：你的 agent 是 5 秒一次代码执行
（E2B 就够）还是 100 ms 一次（那 CreateOS 有价值）。

### 4. ditto.site · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/ditto-site) · 428 votes / 94 comments · **ratio 0.22**

**定位**：任意 URL → 干净的 Next.js 或 Vite 项目，**几分钟**。
Components / tokens / interactions / hover state 都还原。**deterministic
（不用 LLM 猜）**，所以 fast / cheap / consistent。免费 REST API + MCP
server + 开源。

**差异点**：v0、Bolt、Lovable 都是"文字 / 图 → 前端代码"，路径是 LLM
生成，代码质量不稳、每次输出不同。Ditto 走**"页面 → AST → 结构化代码"**
的确定性路线：解析已经渲染出来的 DOM + CSS + 交互，反推组件树，用编译
而非生成得到代码。这跟 htmltodesign / Anima 这类历史工具在概念上一致，
但**免费 + 开源 + MCP** 是新组合——htmltodesign 是订阅制 SaaS，Anima 也
一直是设计工具付费墙。**MCP server** 意味着可以直接被 Claude Code /
Cursor 调用作为"从 UI 学 UI"的工具。

**技术判断**：deterministic 路线的两个技术核心：**1）从渲染后的
computed style + DOM 反推出组件边界**——需要一套启发式（重复模式检测、
盒模型对齐、层级切片），做好了才不会输出扁平 HTML；**2）interaction
状态还原**——hover / focus / active，需要模拟事件抓 style 变化。开源
+ 免费是**用开源做前端流量入口**的经典打法，商业化空间在企业版
（token 系统集成、design system mapping）。

**信号价值**：**"复刻竞品前端"这件事从灰产变成 dev tool**——加上 MCP
后，AI IDE 里可以直接"给我做一个像 X 网站的 landing page"→ Cursor
调 Ditto → 拿到结构化组件。对做 dev tools 团队：**MCP 正在把之前的
SaaS 工具变成 IDE 里的"能力"**——不接 MCP 的工具正在丢入口。

### 5. tterm · ⭐⭐⭐⭐
[PH](https://www.producthunt.com/products/tterm) · 134 votes / 52 comments · **ratio 0.39**

**定位**：**macOS 上专门给 Claude Code 用的驾驶舱**。每个项目一行：
explorer + Claude + file viewer，可以并排堆叠多个 project。**内嵌真实
的 Chromium 浏览器**，**故意不做编辑器**——你只做 diff review + commit。
甚至能"跟 Claude pane 说要加个功能，跑着的 app 自己 hot-reload"（自举）。

**差异点**：Cursor / Zed / Windsurf 都还是"IDE + AI 辅助"的传统包装；
tterm 反过来：**去掉编辑器，只留 review 和 chat**——承认"AI 生成 →
人 review commit"才是 workflow 主线。同期 Aider（CLI）也是"不编辑，
review + commit"的路子，但 Aider 是纯终端；tterm 加了 GUI 布局 +
内嵌浏览器（对前端项目"改 → 立刻在旁边刷新页面"）。**多 project
row**（多个 Claude session 平行跑）是原生假设，跟 Cursor 的单窗口
模型完全不同。

**技术判断**：内嵌 Chromium 是 Electron / Tauri 路线（多半 Tauri，
体积小 + Apple silicon 原生）；"自举 hot-reload"意味着 tterm 本身
用同样的 architecture 跑，Claude 改代码后触发 reload——这只有当 app
用 SwiftUI + Metal Live Reload 或者 web-based UI 才做得动。**故意没
账号 / 没遥测 + 一次性买断（免费给爱好者）**是"反 SaaS 的开发者工具"
姿态，跟同期 Aider、Zed（本地 first）一脉相承。

**信号价值**：**"post-Cursor" 的编码 surface 开始试探**——AI 写代码
量级起来后，人做 review + 分派多个 session 才是瓶颈。tterm 押注的是
"multi-agent 并行工作 + 人负责裁决"，Cursor 押注的是"单 session 提高
写代码质量"。两个假设都会有市场，但决定"编码工具下一代长什么样"的是
tterm 这类结构性重排，不是 IDE 里再加一个 sidebar chat。做 dev tool
方向的团队值得反问：**你的产品是在优化"AI 写代码"还是"人管 AI 写代码"**？
